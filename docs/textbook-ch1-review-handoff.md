## レビュー観点

### (A) ネットワークのプロ目線（実務＋ネスペ午後／作問者目線）
- 技術的正確性: L2/L3/L4・L7の役割混同がないか。各リンクはL2フレームで運ばれ中にIPパケット/TCPセグメントが入る前提が崩れていないか。ルータ(L3SW)でL2が付け替わり宛先IPが維持されるか。ARPが「同一LAN内で次ホップ（出口ルータ）のMACを解決する」文脈で正しいか。デカプセル化・ブロードキャスト・ポート(443)等が正確か。
- 簡略化が**誤概念**になっていないか（例: 「ルータの仕事はMACの付け替えだけ」という言い切りの妥当性、TTLや経路選択の省略の是非、最小構成での図の省略が誤解を生まないか）。
- 用語の適切さ: 「出口ルータ＝デフォルトゲートウェイ」等の言い換えが妥当か。誤解を招く語・不自然な専門用語の使い方はないか。
- ネスペ午後への接続: 構成図の境界・次ホップ・許可条件・障害切り分け・答案化につながる見方になっているか。実務の切り分け順序と整合するか。
- 第1章の範囲（最小構成 PC—L2SW—L3SW—Webサーバ）が、最初の一歩として過不足ないか。

### (B) 学習者目線（応用情報合格・インフラ未経験〜1年目）
- 前提知識なしで読めるか。用語が本文で説明される前に唐突に出ていないか。
- 図の意味が本文で説明され、「どこを見るか」が分かるか。図と本文がかみ合っているか。
- 動く図解が操作しやすく、理解を助けているか（カプセル化の包む/外す、ARPの要求/応答、通信フローのMAC変化/IP不変）。
- 読み物として面白いか、続きを読みたくなるか。淡々としていないか。
- 文章の自然さ: 敬体(ですます)で統一され、**常体/敬体の混在や「AI臭」**（「〜することができます」連発、機械的な箇条書き羅列、冗長な前置き）がないか。
- スマホ体験: **横スクロールがないか**（375pxで確認）、文字が潰れていないか、**死にスペースが多すぎないか**、図＋説明が概ね1画面に収まるか。
- つまずきやすい点が先回りされているか。

## レビュー基準（守られているべき制約）
- スマホ375pxで横スクロール禁止。図＋説明は極力1画面、縦スクロールも最小。
- 図はデータ駆動（座標を持たない論理データ＋画面幅でレイアウト算出）。**無限アニメは使わない**。
- 文体は敬体＋体言止め。常体/敬体混在・AI臭は不可。
- 章は段階的に複雑化（第1章は最小構成でよい）。

## 検証Tips
- スマホ横スクロール判定: 375px幅で `window.scrollTo(99999,0); window.scrollX`（>0なら横スクロールあり）、または `document.documentElement.scrollWidth` vs `clientWidth`。
- スクリーンショットはこれらのページでは取得可能（第1章の図は無限アニメを排除済み）。もし固まる図があれば、それ自体が指摘事項（無限アニメ混入の疑い）。
- 図の大きな編集後に dev の HMR が壊れたら dev サーバを再起動する。
