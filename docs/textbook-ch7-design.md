## 0. 構成図の最終形（この章で育てる形）と一貫アドレス

第6章の「2セグメント＋ルータ」に **2台目のルータ** とサーバLANを足し、3セグメントをルータ2台で相互接続する。

| ゾーン | ネットワーク | 主なノード |

## 1. 導入（図なし）

- 第1章「ルータはあて先IPで道を選ぶ」を思い出す。本章はその「どうやって道を選ぶか」の中身。
- ルータが見るのは **あて先IP**。見て決めるのは **次にどの機器へ渡すか（次ホップ）**。その判断材料が **経路表**。
- 第5章「別VLAN間はL3」・第6章「別セグメント間はルータ経由」が、ここで「ではどう中継するのか」に接続。

## 2. 経路表の読み方（図: packet-flow＋sideTable／graph stack）

- **要点**: ルータは経路表（宛先プレフィックス→次ホップ→出口IF→種別）を持ち、あて先IPと照合して **次ホップ** を決める。次ホップは最終あて先ではなく「次の1歩」。
- **図**: `packet-flow`（**graph stack レイアウト**＝全ノードを1枚）。業務PC・別フロアPCがR1の上、R1—R2が縦のルータ間リンク(/30)、サーバLANのWebサーバがR2の下。`sideTable` に R1 の経路表を出し、R1 を通るステップで一致行を `tableHighlightRow` で強調 → 次ホップ 192.168.99.2(R2) を指す。R2 のステップでは表の中身を R2 の経路表に差し替え、172.16.0.0/24＝直結 を強調。**MAC学習（第5章）と同じ「生きた表」の仕組みを再利用**。種別列は前方参照（OSPF）を避けて外し、宛先・次ホップの2列に集約。
- ステップ案（5）:
  1. 業務PC→R1 リンク: あて先IP 172.16.0.20。PCは自分のネットワーク外なのでデフォルトGW（R1）へ。
  2. R1 ノード: 経路表と照合。172.16.0.0/24 の行に一致 → 次ホップ R2(192.168.99.2)。表で該当行を強調。
  3. R1→R2 リンク: あて先IP 172.16.0.20 は **変わらない**（付け替わるのはL2のMACだけ＝第1章接続）。次ホップへ送る。
  4. R2 ノード: 経路表を R2 のものに差し替え。172.16.0.0/24＝直結 の行を強調 → 直接届ける。
  5. R2→Webサーバ リンク: サーバLANのWebサーバへ到達。
- **図中に明記**: 各ルータの両端IP・経路表の宛先/次ホップ。**説明文任せにしない**。
- **つまずき(callout warn)**: 「次ホップ」は最終あて先ではない。1台のルータは「次の1歩」しか決めない。あて先IPは基本そのまま、L2のあて名（MAC）だけが区間ごとに付け替わる（第1章の回収）。

## 3. ロンゲストマッチ（図: record-table・highlightRow）

- **要点**: 1つのあて先に複数の経路が一致したら、**最も具体的（プレフィックスが長い）行** を選ぶ。
- **図**: `record-table`。あて先 192.168.30.10 に対し、R1 の経路表で 3 行が一致:
  - 0.0.0.0/0（デフォルト・最後の砦）
  - 192.168.0.0/16（集約経路）
  - 192.168.30.0/24（別フロアの直結）← **highlightRow で強調**
- `emphasizeKey: 'prefix'` でプレフィックス長を目立たせ、「長いほど具体的」を視覚化。
- **つまずき**: デフォルトルート 0.0.0.0/0 は「どれにも一致しない時の行き先」＝第1章のデフォルトGW。一番**短い**ので、他に一致があれば負ける。

## 4. 経路の作り方①スタティック（図: record-table）

- **要点**: 管理者が手で1本ずつ書く経路。確実・予測可能だが、構成変更のたび手直しが要り規模に弱い。
- **図**: `record-table`。R1 に手で書いたスタティック経路の例（宛先→次ホップ→種別=static）。サーバLAN宛て・デフォルトルートなど。
- **フローティングスタティック（軽く・callout tip）**: 普段は使わない予備の経路に、わざと大きいメトリック（距離）を付けておき、主経路が消えた時だけ使う「予備の手書き経路」。名前だけ。
- **つまずき**: 台数・拠点が増えると手書きは破綻 → 次の自動（ダイナミック）へ。

## 5. 経路の作り方②ダイナミック / OSPF と冗長2経路（図: packet-flow／graph loop）★新しい見せ方

- **要点**: ルータ同士が経路情報を交換し、構成変化に自動追従するのがダイナミックルーティング。代表が **OSPF**。
  - OSPF＝**リンクステート**: 各ルータが「どこと、どれだけの太さ（帯域）でつながっているか」を配り合い、全員が **同じ全体地図** を持つ。
  - その地図から **コスト（帯域から決まる距離）が最小の経路** を各自が計算（SPF）。**ホップ数ではなく速さ（帯域）で最短**を選ぶのがOSPFの勘所。
  - **エリア**＝大きなネットワークを区切って地図を分割する仕組み。**名前だけ押さえればよい**。
- **構成図の差分（冗長2経路）**: R1 と R2 を **2本の経路** で結ぶ（主経路＝速い1Gのリンク／予備経路＝遅い100Mのリンク、または別ルート）。OSPF は速い方をコスト最小として選ぶ。
- **図**: `packet-flow`（`layout:'graph'` の **loop モード**を再利用）。R1・R2 を縦並び、2本の曲線アークで2経路を描く。**各アークにコスト（帯域）ラベル**を添える（GraphTopology に「ループアークの任意ラベル」を小さく追加。下記§実装メモ）。
  - ステップ1: 2経路とも生きている。OSPF はコスト最小（速い方）を選び、その**アークを強調＋進行方向の矢印**。説明=「帯域からコストを計算し、コスト最小の経路を選ぶ」。
  - ステップ2（冗長の切替＝午後頻出）: 主経路が切れた状態を `blockedLink` で **✕＋破線**。OSPF が再計算し、**予備アークが新しい最短**として強調。説明=「主経路が切れると自動で予備へ切り替わる」。
- **つまずき(callout)**: スタティック＝手動の確実、ダイナミック(OSPF)＝自動だが設計が要る。OSPF は**距離が帯域基準**＝太い線が「近い」。
- **モックで合意してから実装**（この図だけ新しい見せ方）。

## 6. 経路集約と経路選択（図: record-table ×2、ただし密度を抑える）

- **経路集約**: 連続する複数ネットワークを1つのプレフィックスにまとめ、経路表を小さくする。
  - **図**: `record-table`。集約前（192.168.10.0/24・192.168.20.0/24・192.168.30.0/24…の複数行）→ 集約後（192.168.0.0/16 の1行）。`emphasizeKey:'prefix'`。
  - つまずき: まとめすぎると、その範囲の細かい経路が1本に隠れる（実在しないネットワークも含んでしまう）。
- **経路選択（AD→メトリック）**: 同じあて先に複数の「経路源」（直結・スタティック・OSPF）から経路が来たら、**アドミニストレーティブディスタンス（経路源の信頼度＝優先順位）** で選び、同じ経路源なら **メトリック（距離）** で選ぶ。
  - **図**: `record-table`。経路源→AD（直結0 < スタティック1 < OSPF110 …）→説明。直結が最優先＝highlightRow。AD は「名前と順番だけ」。
- **密度配慮**: 2つの record-table を1節に置くが、それぞれ別の主張（まとめる／選ぶ）。本文・caption で重複説明しない。

## 7. 午後での問われ方（図なし or §2図の再掲なし・テキスト＋callout）

- 経路表を読んで「この通信はどのルータのどの行に一致し、次ホップはどこか」を追う（最頻出）。
- ロンゲストマッチで一致行を選ぶ／冗長経路の切替（主経路ダウン時にどこを通るか）／経路集約の読み。
- **技術メモ（本文・callout）**: ルータを越えるたび **TTL が1減る**（ループ防止。0になったら破棄）。到達確認の **ping**・経路確認の **traceroute** は **ICMP** を使う（第19章の障害切り分けで活用）。BGP は第8章。

## 8. まとめ（takeaways）

1. ルータは **経路表＋ロンゲストマッチ** で次ホップ（次の1歩）を選ぶ。あて先IPは基本そのまま、L2のあて名だけ区間ごとに付け替わる。
2. 経路の作り方は **スタティック（手動・確実）** と **ダイナミック / OSPF（自動・変化に追従）**。
3. **OSPF はリンクステート**＝全員が同じ地図を持ち、**コスト（帯域）で最短**を選ぶ。エリアは名前だけ。
4. 冗長な2経路があれば、主経路が切れても**自動で予備へ切替**（午後頻出）。
5. **経路集約**で経路表を小さく、複数経路源は **AD→メトリック** で選ぶ。**デフォルトルート 0.0.0.0/0＝第1章のデフォルトGW**（最後の砦）。

## 実装メモ（コンポーネント拡張は最小限）

- **唯一の拡張候補**: `GraphTopology` の **loop アークに任意ラベル（コスト/帯域）** を描けるようにする。`Topology` か `PacketFlowFigure` に「ループ2本それぞれのラベル」を持たせる小フィールド追加（例: `Topology.loopLabels?: [string, string]`）。型→GraphTopology の loop 描画にラベル text を2つ追加するだけ。新 kind・新 role は不要。
  - もしユーザーが「コストは記録表だけでよい（アーク無ラベル）」を選べば、**拡張ゼロ**で §5 図は loop モードそのまま＋§5 に小さな record-table を添える形に変更。
- それ以外（§2〜4・6）は既存コンポーネントで描ける。types/Renderer 変更なし。
- 検証は第1章水準: 375 / 360 / 1280 横スクロール0、全図の全ステップ送りでボタン不動、`tsc --noEmit` 0、コンソール無警告。
