# 教科書モード 章別 詳細設計

作成日: 2026-06-23

`docs/textbook-curriculum-design.md`（全体マップ）で合意した章順に沿って、各章を **節レベル** まで設計する。実装（本文ドラフト・図のステップ）は本設計をもとに章ごとに行い、第1章で確立した検証フロー（スマホ375/360・全ステップ送り・ボタン固定・横スクロール0・型チェック）を必ず通す。

各章の書式:
- **ねらい / 前章との接続**（なぜこの順か）
- **構成図の差分**（topology ramp のどこを足すか）
- **節構成**（見出し順。各節: 本文の要点 → 図[種類・見せる内容・動き] → つまずき）
- **まとめ（takeaways）** と **技術正確性メモ／午後フック**

**図の選択と画面仕様は `docs/textbook-figure-spec.md` を正とする。** 本書の各節が挙げる図は、図解仕様書の使い分け・カタログに従って確定する。とくに **プロトコルの往復（DHCP DORA／TCP3way／TLS／HTTP要求応答／RADIUS）は `packet-flow` ではなく新図 `sequence`（縦のシーケンス／ラダー）を使う**（順序を主役にする方が圧倒的に読みやすい）。新図 kind は `sequence` / `record-table` / `subnet-calc` / `timeline`。

第1章の「PC→Webの旅」を、機器を増やさずに完結させる。第1部を読み終えると「ブラウザでURLを開いてからページが返るまで」を端から端まで説明できる。

### 第2章　名前解決とアドレス配布（DNS・DHCP）

**ねらい / 接続**: 第1章は「WebサーバのIP(172.16.0.20)を知っている前提」「PCにIPとデフォルトGWが設定済みの前提」で進めた。本章はその**出発前**を埋める。自分の住所をもらう(DHCP)→相手の住所を調べる(DNS)→第1章のARP/TCPへつながる。これで旅に「始まり」ができる。

**構成図の差分**: 内部LANに **DNSサーバ** と **DHCPサーバ** を追加（PC—L2SW—…—ルータ＋内部LANに2サービス）。第1章の図に2台足すだけなので、構成図の連続性を保てる。

**節構成**:
1. **導入**: 第1章は住所を知っている前提で進めた。実際は通信の前に2つの準備がある——「自分の住所(DHCP)」と「相手の住所(DNS)」。
2. **PCが起動して住所をもらう（DHCP）**
   - 要点: 電源を入れたPCはまだ自分のIPを持たない。だから「誰か住所をください」とブロードキャストで探す。もらうのは **IP・サブネットマスク・デフォルトGW・DNSサーバ** の4点セット。→第1章のGW/ネットワーク判定は「ここで配られていた」と接続。
   - 図: `packet-flow`（DORA 4ステップ: Discover→Offer→Request→Ack。Discover/Requestはブロードキャスト、Offer/Ackはサーバ→PC）。ヘッダ表は L2=ブロードキャスト/ユニキャスト、L4=UDP67/68 を見せる。
   - つまずき(callout warn): まだ自分のIPが無いので、最初は全員あて（ブロードキャスト）で探すしかない。
3. **名前からIPを調べる（DNS）**
   - 要点: `www.example.com` を `172.16.0.20` に変える。PCはまず自分に設定されたDNSサーバ（DHCPでもらった）に聞く。DNSサーバが代わりに ルート→TLD→権威 とたどって答えを持ってくる（再帰問い合わせ）。
   - 図: ★新図候補 `dns-hierarchy`（スタブ→キャッシュDNS→ルート/TLD/権威の階層を、問い合わせが降りて答えが返るステップ）。`packet-flow` のトポロジ差し替えで代用可かは実装時判断。
   - つまずき: キャッシュとTTL（一度引いた名前はしばらく覚える）。hostsの存在は軽く。
4. **つながる直前までを通す（第1章への接続）**
   - 要点: DHCPで自分の住所→DNSで相手のIP→（第1章）GWのMACをARP→TCP。第1部の時系列が1本につながる。
   - 図: 第1章の構成図に「①DHCP ②DNS ③ARP ④TCP」の順序を重ねた俯瞰（既存トポロジの再利用）。
5. **午後での問われ方**: 名前解決の失敗切り分け（DNS応答が無い/誤り→名前で通信できないがIP直打ちなら通る）。DHCPスコープ枯渇。
6. **まとめ**。

**takeaways**: 通信の前に「自分の住所(DHCP)」「相手の住所(DNS)」を用意する／DHCPでもらう4点セット(IP・マスク・GW・DNS)／DNSは名前→IPの変換で、ふだんはキャッシュが効く。

**技術メモ/午後**: DHCP=UDP67/68・ブロードキャスト・DORA。DNS=UDP53（大きい応答・ゾーン転送はTCP53）。再帰/反復問い合わせは簡略化（概要のみ、深追いしない）。午後はDNS/DHCP障害の切り分けが頻出。

### 第3章　TCP・UDPとポート番号

**ねらい / 接続**: 第1章で「TCP SYN」「ポート443」を出したが中身は説明していない。本章でコネクションを張る仕組み（3wayハンドシェイク）、UDPとの違い、ポート/5タプルでの通信識別を深掘り。第4章TLS・第8章FW（許可条件）・第7章NATの土台になる。

**構成図の差分**: なし（PCとWebの**会話をズーム**）。構成図は第1章のまま、注目を通信そのものに移す。

**節構成**:
1. **導入**: 第1章の「TCP SYN」は何の始まりだったのか。データを送る前に、まず**つなぐ**作業がある。
2. **コネクションを張る（3wayハンドシェイク）**
   - 要点: SYN→SYN・ACK→ACK の3往復で「送る準備ができたね」を相互確認してからデータを送る。
   - 図: `packet-flow`（PC↔Web、3ステップ。方向が往復することを既存の reverse 表示で見せる）。ヘッダ表に SYN/ACK のフラグと、宛先ポート443・送信元ポート(エフェメラル)。
   - つまずき: ハンドシェイクは「データではなく合図」。シーケンス番号は「どこまで届いたか」の番号、程度に留める。
3. **TCPとUDPの違い**
   - 要点: TCP=届いたか確認・順序保証・再送（確実だが重い）。UDP=確認しない・軽い・速い（取りこぼし許容）。用途の対比（Web/メール=TCP、DNS/VoIP/動画=UDP）。
   - 図: `address-table` 流用の2列比較（TCP/UDP × 信頼性・順序・速度・用途）。
   - つまずき: 「UDPは不便」ではなく「速さ優先の割り切り」。
4. **ポートと5タプル**
   - 要点: 1本の通信は **送信元IP・宛先IP・プロトコル・送信元ポート・宛先ポート** の5つで一意に決まる。宛先ポートでサーバ内のアプリ(443=HTTPS)、送信元ポートで自分側のどの通信かを区別。
   - 図: ★新図候補 `tuple-card`（1本の通信を5タプルのカードで表示）。`address-table` 拡張で代用可。
   - つまずき: 同じPCから同じサーバへ複数同時通信できるのは送信元ポートが違うから。→第7章NAPT・第8章FW許可条件の伏線。
5. **午後での問われ方**: 5タプルで通信を特定、ウェルノウンポート、コネクションの状態（FWのステートフルへ）。
6. **まとめ**。

**takeaways**: TCPは3wayでつないでから送る／TCP=確実・UDP=軽快の使い分け／通信は5タプルで一意。

**技術メモ/午後**: 3way後にデータ、クローズは4way（FINは軽く触れる程度）。エフェメラルポート。状態（ESTABLISHED等）は第8章ステートフルFWで活用。

### 第4章　Web通信の中身（TLS・HTTP）

**ねらい / 接続**: 第1章は「ポート443→HTTPSのアプリへ」で止めた。その封の中身を開く——TCP(第3章)の上にTLS、その中にHTTP。第1部の締めとして「URLを開いてページが返るまで」を完成させる。証明書は第12章PKIへ、TLS終端は第9章プロキシへ伏線。

**構成図の差分**: なし（**ズーム**）。

**節構成**:
1. **導入**: 443の封を開けると何がある？ 下からTCP→TLS→HTTPの入れ子。第1章のOSI上位層に戻ってくる。
2. **TLSハンドシェイクと証明書**
   - 要点: 通信を始める前に「暗号化の取り決め」と「相手が本物かの確認(サーバ証明書)」を行う。暗号化＝盗み見防止、証明書＝なりすまし防止。
   - 図: `packet-flow`（簡略ハンドシェイク: ClientHello→ServerHello+証明書→鍵共有→暗号化開始）。深入りせず流れ重視。
   - つまずき(callout): 「暗号化」と「本人確認」は別の役割。鍵の数学には踏み込まない。
3. **HTTPの要求応答**
   - 要点: 暗号トンネルの中でHTTP。メソッド(GET/POST)→ステータス(200/404)→ヘッダ＋本文。要求と応答の1往復。
   - 図: ★新図候補 `req-res`（要求カードと応答カードの対）。`packet-flow` 2ステップで代用可。
   - つまずき: HTTPSはHTTPが消えるのではなく「TLSで包まれたHTTP」。
4. **1リクエスト完結の全景（第1部まとめ図）**
   - 要点: DHCP→DNS→ARP→TCP3way→TLS→HTTP を1枚で振り返る。第1部で学んだ全段を時系列で接続。
   - 図: ★新図候補 `timeline`（縦タイムライン: 各段で「誰が・何を」）。第1部の総まとめ。
5. **午後での問われ方**: HTTPS構成、証明書の配置、リバースプロキシでのTLS終端（第9章）。
6. **まとめ**。

**takeaways**: HTTPSはTCPの上のTLSの中のHTTP／暗号化(盗み見防止)と証明書(なりすまし防止)は別役割／第1部でURLを開く全段がつながった。

**技術メモ/午後**: TLSは流れ重視で簡略（1.3前提でよいが版番号には深入りしない）。証明書チェーン詳細は第12章。HTTPは1.1ベースで要求応答の形を押さえる。

## 第2部　ネットワークの土台を組む（構成図を育てる）

### 第5章　L2スイッチング・VLAN・STP
**ねらい/接続**: 第1章の「L2SWはMACを見て中継」の中身を開く。MAC学習・VLAN・STP。ここから構成図が横に育つ。
**構成図の差分**: 内部LANを L2SW×2＋VLAN（VLAN10=業務 / VLAN20=管理）に展開。SW間はトランク、冗長リンク1本をSTP題材に。`nodes`: pc1(VLAN10) / pc2(VLAN20) / sw1 / sw2 / ルータ。`links`: pc-sw, sw1-sw2 ×2(冗長), sw-ルータ。
**節構成**:
1. 導入: 第1章のL2SWを分解する。
2. MAC学習と転送: 送信元MACとポートを `MACアドレステーブル` に学習。宛先MACが分かればそのポートだけ、不明ならフラッディング。図: `record-table`(MAC→ポート) ＋ `packet-flow`(学習→ユニキャスト転送 2-3ステップ)。つまずき: 最初はフラッディング、覚えたらユニキャスト。
3. VLANで分ける: 1台のSWを論理分割。**VLAN=ブロードキャストドメイン**。アクセスポート/トランク(802.1Qタグ)。図: `packet-flow`(ブロードキャストが同VLAN内だけに広がる/別VLANに出ない)。つまずき: 別VLAN間はL3を通らないと不可→第6章。
4. ループとSTP: 冗長リンクはループ→ブロードキャストストーム。STPが片ポートをblockして論理的に1本に。ルートブリッジ/BPDUは軽く。図: `packet-flow`/`topology`(片リンクをblock表示、障害時は復活)。つまずき: 物理2本でも論理1本。
5. 午後: VLAN設計(業務/管理分離)、トランク、ループ事故。
6. まとめ。
**takeaways**: SWはMACを学習し必要ポートだけへ／VLAN=ブロードキャストドメインの論理分割／STPは冗長リンクのループを論理1本に。
**技術メモ/午後**: 802.1Q、ネイティブVLAN(軽)、RSTP(名前のみ)。別VLAN間=L3必須（第6章接続）。**新規**: `record-table`。role追加なし。

### 第6章　IPアドレス設計・サブネットとルーティング
**ねらい/接続**: 第1章の「ネットワーク部で同一/別を判定」を **正式化**。サブネット計算・経路表・ロンゲストマッチ・スタティック→OSPF。第5章の別VLAN間通信(L3)を担う。
**構成図の差分**: セグメント追加（192.168.30.0/24）。ルータ(L3SW)が 10/30/サーバLAN の経路を持つ。
**節構成**:
1. 導入: 第1章で先送りした「ネットワーク部の正体」。IP=ネットワーク部＋ホスト部。
2. サブネットの計算: /24=255.255.255.0。先頭24ビット=ネットワーク、残り8=ホスト。ネットワークアドレス/ブロードキャスト/ホスト範囲。図: `subnet-calc`(192.168.10.10 ＋ /24、ネットワーク部=緑/ホスト部=グレー。/24→/26で境界が動く)。つまずき: 先頭(ネットワークアドレス)と末尾(ブロードキャスト)は端末に使えない。
3. ルータは経路表で次を選ぶ: 宛先のネットワーク部を `経路表` と照合し次ホップ/出口IFを決める。**ロンゲストマッチ**。図: `record-table`(宛先プレフィックス→次ホップ→IF、一致する最長を強調) ＋ `packet-flow`(経路表で次ホップを選ぶ1ステップ、第1章の付け替えと接続)。つまずき: デフォルトルート 0.0.0.0/0 ＝第1章のデフォルトGW。
4. 経路はどう作る: スタティック(手動)とOSPF(経路を交換し最短コストを自動計算)。OSPFは概要(リンクステート/コスト/エリアは名前程度)。図: `topology`(2ルータがOSPFで経路を学び合う)。つまずき: スタティック=確実だが手間、OSPF=自動だが設計要。
5. 午後: サブネット分割、経路表読み、次ホップ、ロンゲストマッチ(頻出)。
6. まとめ。
**takeaways**: IP=ネットワーク部＋ホスト部／ルータは経路表＋ロンゲストマッチで次ホップ／デフォルトルート=第1章のGW／OSPFは経路の自動交換(概要)。
**技術メモ/午後**: サブネット計算は午後必須。OSPFは「自動で最短を学ぶ」レベル。**新規**: `subnet-calc`、`record-table`(再利用)。

### 第7章　インターネット接続・NAT・BGP
**ねらい/接続**: 第6章の経路を「外」へ。プライベート/グローバル、NAPT、AS/BGP。第1章「宛先IPは基本変わらない」の **例外＝NAT** をここで回収。
**構成図の差分**: インターネット境界を追加（境界ルータ＋グローバル203.0.113.1＋internet）。
**節構成**:
1. 導入: 第6章までは社内。社外へ出る仕組み。
2. プライベートとグローバル: 社内=プライベート(192.168/172.16/10.)、世界で通用=グローバル。そのままでは外に出られない。図: `address-table`(プライベート範囲/グローバル)。
3. NAT/NAPT: 境界で送信元プライベートIP:ポート→グローバルIP:ポートに書き換え、`変換表` で戻りを対応づけ。多数を1グローバルで共有。図: `packet-flow`(境界ノードで送信元IP:portを書き換え status=change、第1章の付け替えの発展) ＋ `record-table`(NAT変換表)。つまずき: 外から入るには静的NAT/ポートフォワード(第8章公開サーバへ)。
4. AS/BGP(概要): インターネット=AS(自律システム)の集まり。AS間経路をBGPで交換。自社はISPのASにつながる。図: `topology`(自AS—ISP AS—他AS)。深入りしない。
5. 午後: NAT変換(送信元/宛先の変化)、グローバル/プライベート、公開サーバの戻り経路。
6. まとめ。
**takeaways**: 社内=プライベート/社外=グローバル／NAPTで多数を1つに・変換表で戻す／インターネット=ASの集まり・BGPで経路交換(概要)。
**技術メモ/午後**: NAPT(PAT)/静的NAT。第1章「宛先IPは基本不変」の例外がNAT、と明示(回収)。**新規**: `record-table`(再利用)。

## 第3部　現実の構成へ（境界・拡張・冗長：午後スケール）

### 第8章　セキュリティ境界・ファイアウォール・DMZ
**ねらい/接続**: 第7章で外とつながった。次は「誰を通し誰を止めるか」。FWの許可条件(5タプル=第3章)、ステートフル、DMZ。**午後の最重要構造=境界**。構成図が三層(内部/DMZ/外部)に。
**構成図の差分**: 境界にFW挿入。サーバLANをDMZ(172.16.0.0/24)としてFW配下に再配置。`zones`: 内部LAN / DMZ / インターネット。
**節構成**:
1. 導入: つながった次は「守る」。全部通したら危険。境界で選別。
2. FWは何で許す/止める: ルール=5タプル(第3章)で許可/拒否、原則拒否(デフォルトdeny)。図: `record-table`(FWルール: 番号/送信元/宛先/ポート/動作) ＋ `packet-flow`(FWノードで `verdict`=pass/block。許可される通信と遮断される通信の2例)。つまずき: 上から順に評価、最後はdeny。
3. ステートフル: 行きを許可すると戻りは状態で自動通過。戻り用ルールは要らない。図: `packet-flow`(行き=ルール照合、戻り=状態で通過)。つまずき: 第3章のコネクション状態がここで効く。
4. DMZ: 公開サーバ(Web/メール)を内部と分けた区画に。外→DMZは可、外→内部は不可。図: `topology`(三層、外→DMZ=可/外→内部=不可をハイライト)。つまずき: 公開サーバが破られても内部に波及させないため。
5. 午後: 境界の特定、許可条件の読解、通信可否の根拠(最重要)。
6. まとめ。
**takeaways**: FWは5タプルのルールで原則拒否・必要だけ許可／ステートフルで戻りは自動／DMZで公開サーバを内部から隔離。
**技術メモ/午後**: ステートフルインスペクション、デフォルトdeny、DMZ。NGFW/UTM(名前のみ)。午後の通信可否問題の土台。**新規**: `record-table`(再利用)、`packet-flow` の `verdict` 拡張。role: firewall(既存)。

### 第9章　ロードバランサ・プロキシ・CDN
**ねらい/接続**: 公開Webのアクセス増を捌く。LBで分散、プロキシで代理、CDNで近くから配信。第4章HTTP/TLS・第8章境界の上に乗る。
**構成図の差分**: DMZのWebを LB(VIP 172.16.0.10)＋Webサーバ×2 に。出口にフォワードプロキシ、外部にCDN。**新role**: `lb` / `proxy`。
**節構成**:
1. 導入: アクセス集中をどう捌くか。
2. ロードバランサ: 代表IP(VIP)で受け裏の複数サーバへ振り分け、ヘルスチェックで死んだ台は外す。L4(IP/ポート)とL7(URL/中身、TLS終端)。図: `packet-flow`(クライアント→LB(VIP)→プール内1台) ＋ `address-table`(L4 vs L7)。つまずき: 利用者に見えるのはVIPだけ。
3. プロキシ: リバース=サーバ側の代理(TLS終端・キャッシュ)、フォワード=クライアント側の代理(社内出口で検査/制御)。図: `topology`(リバース=サーバ前/フォワード=社内出口、向きの違い)。つまずき: 「どちら側の代理か」で役割が逆。
4. CDN: 利用者の近くのエッジにキャッシュし近くから返す。オリジンの負荷と距離を減らす。図: `topology`(利用者—近いエッジ—オリジン)。
5. 午後: LBの振り分けと戻り経路、TLS終端位置、リバース/フォワードの区別。
6. まとめ。
**takeaways**: LBはVIPで受け複数へ分散＋ヘルスチェック／L4=アドレス・L7=中身／リバース=サーバ側/フォワード=クライアント側／CDNは近くから配信。
**技術メモ/午後**: VIP、ヘルスチェック、L4/L7、TLS終端(第4章接続)、セッション維持(軽)。**新規**: role `lb` / `proxy`。

### 第10章　冗長化・可用性（と無停止更改）
**ねらい/接続**: 1台壊れたら止まる構成を二重化で止まらなくする。VRRP・リンクアグリ・フェイルオーバー・無停止更改。主要機器(FW/ルータ/LB)を冗長ペアに。
**構成図の差分**: 境界ルータ/FW/LBを2台ペアに。VRRP仮想IP。リンクを2本束ねる(LAG)。
**節構成**:
1. 導入: 単一障害点(SPOF)を無くす。どこが1個だと止まるか。
2. 機器の冗長とVRRP: ルータ/FWを2台、普段はアクティブ、仮想IP(VRRP)を共有。アクティブ故障でスタンバイが仮想IPを引き継ぐ。図: `packet-flow`(2状態: 正常=アクティブ経由 / 故障=アクティブを `down` 表示し経路がスタンバイへ迂回)。つまずき: 端末のGWは仮想IPのまま、中身が切替わる(第1章GW接続)。
3. リンク/経路の冗長: リンクアグリ(LACPで複数物理リンクを1本に束ね帯域＋冗長)、経路の冗長。図: `topology`(2本を束ねる/片方故障で残りが運ぶ)。
4. 無停止更改: 冗長があるから片系ずつ切離→更改→戻すで止めずに交換。図: `timeline`(片系切離→更改→戻す→もう片系)。つまずき: 更改中は冗長が一時的に無い=リスク時間。
5. 午後: SPOFの指摘、切替の仕組み(VRRP)、切替時間。
6. まとめ。
**takeaways**: 冗長化はSPOFを無くす／VRRPは仮想IPをアクティブ/スタンバイで引き継ぐ／LAGはリンクを束ねる／冗長があれば無停止更改ができる。
**技術メモ/午後**: VRRP(HSRPは名前のみ)、LACP、フェイルオーバー。第1章GWの「中身が切替わる」接続。**新規**: `packet-flow` の `down` 拡張、`timeline`(再利用)。

### 第11章　拠点間接続・VPN・WAN・SD-WAN
**ねらい/接続**: 本社と支社をつなぐ。IPsec VPN(暗号トンネル)・WAN・SD-WAN。**第1章のカプセル化が「トンネル=パケットをパケットで包む」として効く**。構成図に第2拠点。
**構成図の差分**: 第2拠点(192.168.20.0/24)を追加。本社ルータ—インターネット/WAN—支社ルータをVPNトンネルで接続。
**節構成**:
1. 導入: 離れた拠点を安全につなぐ。専用線は高い→インターネット経由で安全に。
2. IPsec VPN: 拠点間トンネルで、元のIPパケットを暗号化して新しいIPで包んで送る(第1章カプセル化の応用)。受け側で開いて取り出す。図: `encap`(元パケットをIPsecで包む/開く) ＋ `topology`(本社—トンネル—支社、トンネル区間ハイライト)。つまずき: トンネルの中はもう1枚IPがある(二重)。
3. WANとSD-WAN: WAN=拠点間広域回線(専用線/IP-VPN/インターネット)。SD-WAN=複数回線をソフトで束ね品質/コストで自動選択するオーバーレイ。図: `topology`(複数WAN回線をSD-WANが束ね最適経路)。
4. 午後: 拠点間経路、トンネルの両端、暗号区間、戻り経路。
5. まとめ。
**takeaways**: VPNは元パケットを包んで暗号トンネルで送る(カプセル化の応用)／WANは拠点間回線／SD-WANは複数回線をソフトで束ね最適化。
**技術メモ/午後**: IPsec(トンネルモード)、IKE(名前のみ)、IP-VPN/インターネットVPN、オーバーレイ。第1章encap・第16章VXLANと接続。**新規**: `encap`(再利用)。

### 第12章　認証・認可・SSO・PKI
**ねらい/接続**: 「誰が」「何をしてよいか」を確かめる。認証と認可の区別、RADIUS、SSO、証明書チェーン/PKI(第4章TLS証明書の発展)。第13章無線802.1Xの土台。
**構成図の差分**: 認証サーバ(RADIUS, role=server/auth) 192.168.10.40 を内部に。証明書はオーバーレイ(論理)。
**節構成**:
1. 導入: 第4章で「サーバ証明書で本物か確認」した。今度は利用者側も確かめる。認証と認可は別物。
2. 認証と認可: 認証=あなたは誰か(本人確認)、認可=何をしてよいか(権限)。順は認証→認可。図: `address-table`(認証 vs 認可: 問い/手段/例)。つまずき: ログインできる(認証)とアクセス権がある(認可)は別。
3. RADIUSと集中認証: 各機器がIDを個別管理せず認証サーバ(RADIUS)に問い合わせて一元化。図: `packet-flow`(利用者→認証装置→RADIUS→可否 3-4ステップ)。第13章802.1Xの素地。
4. SSOと証明書/PKI: SSO=一度の認証で複数サービス。証明書チェーン=サーバ証明書←中間CA←ルートCAの署名の連鎖で本物を保証(PKI)。図: `timeline`/`encap` 流用(ルートCA→中間→サーバ証明書のチェーン、第4章TLSと接続)。つまずき: 信頼の起点はルートCA。
5. 午後: 認証フロー、RADIUS、証明書の配置/有効期限。
6. まとめ。
**takeaways**: 認証(誰か)と認可(何をしてよいか)は別／RADIUSで認証を一元化／SSOは一度で複数／証明書はチェーンで本物を保証(PKI)。
**技術メモ/午後**: 認証/認可、RADIUS、SAML/OIDC(名前のみ)、証明書チェーン、失効(CRL/OCSPは名前のみ)。第4章TLS・第13章802.1X接続。**新規**: チェーン図は `timeline`/`encap` 流用で判断。

### 第13章　無線LAN
**ねらい/接続**: 第5章VLAN・第12章認証(RADIUS)の上に「電波という共有媒体」を載せる。AP/WLCで束ね、802.1Xで認証してVLANに載せる。第12章のRADIUSがここで稼働。
**構成図の差分**: 内部LANに AP×2（`ap`）＋ WLC（無線コントローラ、role=ap or server）。APはトランクでVLANに接続。端末→AP→SW。
**節構成**:
1. 導入: ケーブルが電波になると何が変わる——共有媒体・隠れ端末・認証。
2. AP/WLCの役割: APは電波↔有線の変換、WLCが複数APを集中管理(チャネル/出力/ローミング)。SSID=論理ネットワーク名をVLANに対応づけ。図: `topology`(内部LANに AP×2＋WLC)。つまずき: SSIDとVLANの対応。
3. 無線は衝突回避が難しい: 電波は同時に1つ(半二重)、相手が見えない隠れ端末。CSMA/CA(送る前に空き確認＝衝突回避)。図: `sequence`(キャリアセンス→送信→ACK)。つまずき: 有線CSMA/CD(検出)に対し無線はCA(回避)。
4. 無線の認証(802.1X): 端末(サプリカント)→AP(オーセンティケータ)→RADIUS。第12章のRADIUSが稼働、OKで対応VLANへ。図: `sequence`(3者: 端末｜AP｜RADIUS、EAPは簡略)。つまずき: 個人認証(802.1X)とパスフレーズ(PSK)の違い。
5. ローミング: 移動でAPが切り替わってもWLCが引き継ぎ切れない。図: `packet-flow`/`topology`(端末が AP1→AP2、WLCが引き継ぎ)。
6. 午後: 無線設計(チャネル/VLAN/認証)、ローミング、電波干渉。
7. まとめ。
**takeaways**: APは電波↔有線・WLCが集中管理／無線はCSMA/CAで衝突回避／802.1Xは端末-AP-RADIUS(第12章)／SSIDをVLANに対応。
**技術メモ/午後**: CSMA/CA、隠れ端末、802.1X/EAP、WPA2/3、ローミング。**新規role**: `ap`。図は既存＋`sequence`。

### 第14章　IPv6
**ねらい/接続**: 第6章IPv4アドレス・第1章ARPの対比で入る。枯渇への根本解決＝IPv6。ARPの代わりにNDP、自動設定(SLAAC)、移行はデュアルスタック。
**構成図の差分**: 既存ノードにIPv6を重ねる(デュアルスタック)。構成図は同じでアドレスが2系統(オーバーレイ＝論理)。
**節構成**:
1. 導入: なぜIPv6——IPv4は枯渇(NATで延命＝第7章)。根本解決がアドレスを増やすIPv6。
2. IPv6アドレスの形: 128ビット、16進＋コロン、`::`省略、プレフィックス/64。図: `address-table`(IPv4 vs IPv6: 長さ/表記/NAT前提) ＋ 任意で`subnet-calc`流用(プレフィックス/インターフェースID色分け)。つまずき: 桁は多いが「前半=ネットワーク・後半=ホスト」は同じ。
3. ARPの代わりにNDP: 同一リンクのMAC解決はNDP(近隣探索, ICMPv6)。第1章ARPの役割をIPv6ではNDPが担う。図: `sequence`(近隣要請NS→近隣告知NA、第1章ARPと並べる)。つまずき: 「ARP=IPv4／NDP=IPv6」。
4. 自動設定(SLAAC)と移行: ルータ広告(RA)を聞いて自分でアドレスを作る。当面はIPv4/IPv6併存(デュアルスタック)。図: `timeline`(RA→アドレス生成)。つまずき: DHCPなしでも設定できる。
5. 午後: IPv6移行、デュアルスタック、NDP、表記。
6. まとめ。
**takeaways**: IPv6は枯渇の根本解決／前半ネットワーク・後半ホストはIPv4と同発想／ARP→NDP(第1章対応)／SLAACで自動設定・移行はデュアルスタック。
**技術メモ/午後**: 128bit、`::`、/64、NDP(NS/NA/RA)、SLAAC、デュアルスタック。NAT64は名前のみ。図は既存流用。role追加なし。

### 第15章　メール
**ねらい/接続**: 第2章DNS(MX)・第8章DMZ・第4章TLSの合流点。配送(SMTP)と受信(IMAP)は別経路、なりすまし対策(SPF/DKIM/DMARC)はDNSを使う。
**構成図の差分**: DMZにメールサーバ(`mail`) 172.16.0.25、外部に相手MX。
**節構成**:
1. 導入: メールはWebと違い「ためて転送」。送信と受信は別の仕組み。
2. 送信の配送(SMTP): 送信者→自社メールサーバ→(DNSでMX解決)→相手メールサーバ、サーバ間をSMTPでバケツリレー。図: `sequence`(送信者｜自社MTA｜相手MTA) ＋ `topology`(DMZメール→インターネット→相手MX)。つまずき: 送信はサーバ間リレー、受信は別。
3. 受信: 届いたメールを利用者がIMAP/POPで取りに行く。図: `sequence`(利用者｜メールサーバ)。
4. なりすまし対策: SPF=送信元IPをDNSで照合、DKIM=署名で改ざん検知、DMARC=SPF/DKIMの結果で方針をDNSに公開。第2章DNSが土台。図: `record-table`(SPF/DKIM/DMARCの役割対比)。つまずき: 3つは役割分担(IP/署名/方針)。
5. 午後: 送信ドメイン認証、メール経路、DMZ配置。
6. まとめ。
**takeaways**: 送信はSMTPでサーバ間リレー(MXはDNS)、受信はIMAP/POP／SPF=IP・DKIM=署名・DMARC=方針／DNSがメールの土台。
**技術メモ/午後**: SMTP(25/587)、IMAP/POP、MX、SPF/DKIM/DMARC、STARTTLS。**新規role**: `mail`。図は既存＋`sequence`/`record-table`。

### 第16章　仮想化・クラウド（SDN・VXLAN・EVPN）
**ねらい/接続**: 第5章VLAN・第11章オーバーレイの発展。サーバ仮想化、クラウド(VPC)、物理を意識しない論理ネット(VXLAN)、制御(SDN/EVPN)。第1章カプセル化・第11章トンネルの集大成。
**構成図の差分**: クラウドゾーン(VPC 10.0.0.0/16, `cloud`)を追加。VXLANで拠点—クラウドを論理接続。
**節構成**:
1. 導入: サーバもネットワークも「ソフトで作る」。物理の上に論理を重ねる。
2. サーバ仮想化とクラウド: 1台の物理に複数VM、仮想スイッチで接続。クラウド=VPCという自分専用の仮想ネット。図: `topology`(物理サーバ内にVM＋仮想SW、クラウドにVPC)。つまずき: VMもIP/MACを持つ端末。
3. オーバーレイ(VXLAN): 物理網(アンダーレイ)の上にL2を延伸する論理トンネル。元フレームをUDPで包んで運ぶ(第1章/第11章の発展)。図: `encap`(元フレームをVXLAN/UDP/IPで包む) ＋ `topology`(拠点—アンダーレイ—クラウド)。つまずき: VLANは4096、VXLANは大規模(VNI)。
4. SDNとEVPN(概要): SDN=制御を集中(コントローラが経路を配る)。EVPN=VXLANの制御plane(MACの居場所を配布)。概要のみ。図: `topology`(コントローラが各機器へ設定配布)。
5. 午後: クラウド接続、オーバーレイ、VPC/拠点接続。
6. まとめ。
**takeaways**: 仮想化で物理1台に多数VM／クラウドはVPC／VXLANは物理の上にL2を延ばすオーバーレイ(包んで運ぶ)／SDN/EVPNが制御(概要)。
**技術メモ/午後**: VM/仮想SW、VPC、VXLAN(VNI/UDP4789)、アンダーレイ/オーバーレイ、SDN、EVPN。第1章encap・第11章トンネル接続。図は`encap`/`topology`流用。role: cloud(既存)。

### 第17章　VoIP・QoS・マルチキャスト
**ねらい/接続**: 第3章UDP(VoIP=UDP/RTP)・第5/6章の上。音声は遅延に弱い→QoSで優先。一斉配信はマルチキャスト。
**構成図の差分**: 内部LANにIP電話(`phone`) 192.168.10.5x。QoSは経路上のマーキング(論理)。
**節構成**:
1. 導入: 音声/映像はデータと何が違う——遅れ・ゆらぎに弱い、取りこぼしは許容(UDP)。
2. VoIPの仕組み: 音声をパケットに(RTP/UDP)、呼制御(SIP)で接続。図: `sequence`(電話A｜電話B: SIPで接続→RTPで音声)。つまずき: 取りこぼしてもやり直さない(UDP)、遅延が致命的。
3. QoS(優先制御): 混雑時に音声を優先。優先度を付け(マーキングDSCP)、優先キューで処理。図: `packet-flow`/`topology`(混雑リンクで音声が先に通る) ＋ `record-table`(クラス分け例)。つまずき: 帯域を増やすのでなく優先順位を付ける。
4. マルチキャスト: 同じデータを多数へ一斉(配信は1本、必要な所で複製)。図: `topology`(送信1→分岐で複製→受信者群、IGMPで参加)。つまずき: ユニキャスト/ブロードキャスト/マルチキャストの別。
5. 午後: QoS設計、VoIPの遅延/帯域、マルチキャスト参加。
6. まとめ。
**takeaways**: VoIPは音声をUDP/RTPで運びSIPで呼制御／QoSは優先順位で音声を守る／マルチキャストは参加者だけへ一斉配信。
**技術メモ/午後**: RTP/UDP、SIP、DSCP/マーキング、優先キュー、IGMP。**新規role**: `phone`。図は既存＋`sequence`/`record-table`。

### 第18章　IoT・LPWA（概要）
**ねらい/接続**: 小章。多数の小型機器、省電力・広域(LPWA)。**概要のみ・深追いしない**。
**構成図の差分**: IoTゲートウェイ(`iot`)＋デバイス群(軽量)。
**節構成**:
1. 導入: 大量の小さな機器をどうつなぐ——電池が長持ち・遠くまで届く通信。
2. IoTの通信像: デバイス→ゲートウェイ→クラウドへ集約。軽量プロトコル(MQTT)で少量データを頻繁に。図: `topology`(デバイス群→GW→クラウド)。つまずき: 多数・省電力・小データ。
3. LPWA(概要): 省電力で広域(数km)・低速。位置づけだけ。図: `address-table`(WiFi/携帯/LPWA: 距離・速度・電力)。
4. 午後: 軽め(概要把握)。
5. まとめ。
**takeaways**: IoTは多数・省電力・小データ／デバイス→GW→クラウドに集約／LPWAは省電力広域(概要)。
**技術メモ/午後**: MQTT、LPWA(LoRaWAN/NB-IoTは名前のみ)。深追いしない。**新規role**: `iot`。図は既存流用。

## 第4部　運用と総合

### 第19章　運用監視・障害切り分け
**ねらい/接続**: 全章を「運用」視点で束ねる。監視(SNMP/syslog/フロー)、切り分けの段取り(第1章の層分けを実務化)。
**構成図の差分**: 監視サーバ(`monitor`)を内部に。全機器を監視対象として俯瞰。
**節構成**:
1. 導入: 作った後どう守る——異常に気づく(監視)、起きたら切り分ける。
2. 監視の三本柱: SNMP(状態をポーリング/トラップ)、syslog(ログ収集)、フロー(NetFlow/sFlowで通信量)。図: `topology`(監視サーバが各機器から収集) ＋ `record-table`(SNMP/syslog/フローの役割対比)。つまずき: 状態/ログ/量で道具が違う。
3. 障害切り分けの段取り: 層で分ける(第1章)。物理→L2→L3→上位、近い順に。ping(到達)→経路→名前解決→ポート。図: `timeline`(切り分け手順)。つまずき: 闇雲でなく層と経路で順に絞る。
4. 午後: 障害切り分け、監視指標、ログの読み。
5. まとめ。
**takeaways**: 監視はSNMP(状態)/syslog(ログ)/フロー(量)／切り分けは層と経路で順に絞る(第1章の実務化)。
**技術メモ/午後**: SNMP(ポーリング/トラップ/MIB)、syslog、NetFlow/sFlow、ping/traceroute。第1章の層分けを実務手順に。**新規role**: `monitor`。図は既存＋`timeline`/`record-table`。

### 第20章　午後問題の読み方・総合演習
**ねらい/接続**: 積み上げた全体図＝午後相当で読解の型を総合演習。**新概念なし・統合**。topology ramp の到達点。
**構成図の差分**: これまでの全要素を集約した「午後相当の全体図」(内部LAN/DMZ/インターネット/第2拠点/クラウド/冗長/監視)。
**節構成**:
1. 導入: ここまでの全部が1枚に。午後の構成図はこう読む。
2. 構成図の読み方(型): まず境界(内部/DMZ/外部/拠点/クラウド)→各通信の経路と次ホップ→許可条件(FW)→冗長。第1章「境界と次に渡す相手」の完成形。図: `topology`(全体図に、読む順①境界②経路③許可④冗長をハイライト)。
3. 通信を1本追う(総合): 例「社外→Web公開サーバへHTTPS」を DNS→境界→FW許可→LB→Web と全章接続で追う。図: `packet-flow`(全体図上を端から端まで)。つまずき: 1本の通信に全章が関わる。
4. 設問の答え方: 「なぜ通信できないか」「どの機器が何を見るか」を層と境界で根拠化。図: `timeline`/`record-table`(切り分け→根拠→答案化)。
5. まとめ(教科書全体の締め): 構成図が読めるようになった。
**takeaways**: 午後は「境界→経路→許可→冗長」で読む／1本の通信に全章が関わる／答えは層と境界で根拠化。
**技術メモ/午後**: 総合（全章を相互参照）。図は`topology`(全体図)/`packet-flow`/`timeline`。新規なし。
