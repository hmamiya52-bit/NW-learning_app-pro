# 図解で学ぶ教科書 章別詳細設計書

## 1. この詳細設計書の目的

「図解で学ぶ教科書」は、初学者が最初に読む教材である。したがって、本文を書き始める前に、章ごとの学習導線、図解、動く図解、試験対策との接続を詳細設計として固める。

この文書を、今後の教科書本文実装の基準にする。実装済みの本文がこの詳細設計と矛盾する場合は、実装済み本文を優先しない。

## 2. 設計原則

### 2.1 最初に全体地図を出す

各章の冒頭では、いきなり個別技術に入らない。最初に「この章はネットワーク全体のどこを見ているのか」を示す。

第1章では、必ずOSI参照モデルから入る。L2、MAC、ARP、VLANを最初の話題にしない。初学者にとっては、L2が何の文脈で出てくるかが分からないためである。

### 2.2 静止画で構造、動く図で時間変化を説明する

静的な図は、関係性や構造を理解させるために使う。

動く図は、次のような時間変化を理解させるために使う。

- データにL4ヘッダ、L3ヘッダ、L2ヘッダ/トレーラが付く
- 中継機器を通るたびに、どのヘッダを見ているかが変わる
- ルータでL2を外し、L3を見て、次のリンク用のL2を付け直す
- サーバ側でL2、L3、L4の順に外され、アプリケーションデータが渡される

### 2.3 原則として全ての節に図解を置く

教科書の各節には、原則として最低1つの図解を置く。文章だけで概念を説明しない。

特に、次のような概念は文章だけで説明してはいけない。

- OSI参照モデル
- カプセル化、デカプセル化、再カプセル化
- MACアドレス、IPアドレス、ポート番号の使い分け
- ARP要求とARP応答
- VLANとブロードキャストドメイン
- 構成図上のL2境界、L3境界

図解には、必ず「この図で何を見るか」を添える。図があるだけで説明がない状態も不可とする。

例外は、短いまとめ節や前節の図を明確に再利用する節に限る。その場合でも、本文中で「前の図のどこを見直すか」を明記する。

### 2.4 誤解を招く表現を禁止する

次の表現は禁止する。

- 別ネットワークにはIPパケットを送る
- 同一セグメントではフレーム、別ネットワークではパケットを送る
- ルータがL4ヘッダを付け直す
- L2SWがIPアドレスを見て転送する
- PCが別ネットワーク上のWebサーバのMACアドレス宛てに直接送る

正しくは、各リンク上ではL2フレームとして運ばれ、その中にIPパケットが入る。ルータは受信したL2フレームをデカプセル化し、IPヘッダを見て経路判断し、次のリンク用のL2フレームに再カプセル化して送る。

### 2.5 「試験対策」と「実務理解」を同じ説明でつなぐ

試験対策だけに寄せて用語暗記にしない。実務だけに寄せて試験の設問根拠が見えない説明にもしない。

各節で、次の2つを必ず整理する。

- ネスペで問われやすい見方
- 実務で障害切り分けや設計に効く見方

## 3. 章詳細設計の標準フォーマット

各章を実装する前に、次の項目を埋める。

- 章タイトル
- 対象読者の開始状態
- 章を読み終えた後の到達状態
- この章で扱う範囲
- この章で扱わない範囲
- 節構成
- 各節の目的
- 各節の本文要点
- 各節の図解仕様
- 動く図解仕様
- ネスペ午後問題への接続
- 実務への接続
- 誤解させてはいけない表現
- セルフレビュー観点

## 4. 第1章 詳細設計

### 4.1 章タイトル

OSI参照モデルと通信の全体像

既存の「レイヤ1〜3基礎」という名前は、内容実装上の都合が強い。初学者が最初に読む章としては、まずOSI参照モデルと通信全体の見方を作り、その中でL1〜L3、L4、MAC、IP、ARP、VLANを位置付ける。

### 4.2 対象読者の開始状態

- 応用情報技術者レベルの用語は一度見たことがある
- OSI参照モデルの7階層名は暗記したことがあるかもしれない
- ただし、通信時にヘッダがどの順番で付くかは曖昧
- MACアドレス、IPアドレス、ポート番号の使い分けが曖昧
- L2SW、ルータ、サーバがそれぞれ何を見て動くかが曖昧

### 4.3 章を読み終えた後の到達状態

- OSI参照モデルを、暗記表ではなく「通信を見るための地図」として説明できる
- アプリケーションデータにL4、L3、L2の情報が順に付くことを説明できる
- L2SW、ルータ、サーバがそれぞれどのヘッダを見るかを説明できる
- ルータではL2が付け替わるが、IPパケットは宛先IPを保って進むことを説明できる
- ARPが「IP通信の前に急に出る謎の仕組み」ではなく、L2フレームを作るために必要になる仕組みだと理解できる

### 4.4 この章で扱う範囲

- OSI参照モデルの全体像
- TCP/IP通信におけるL4、L3、L2のカプセル化
- クライアント、L2SW、ルータ、サーバの役割
- MACアドレス、IPアドレス、ポート番号の見ている範囲
- ARPとデフォルトゲートウェイ
- VLANとブロードキャストドメインの入口

### 4.5 この章で扱わない範囲

- TCPの3ウェイハンドシェイクの詳細
- HTTP、TLSの詳細
- OSPF、BGPなどの動的ルーティング
- NATの詳細
- VLAN設計の詳細
- STP、リンクアグリゲーション

これらは後続章で扱う。

## 5. 第1章の節構成

### 5.0 第1章の図解配置方針

第1章では、全ての節に図解を置く。第6節は動く図解を中心にするが、他の節も静的図または小さな補助図を必ず持つ。

| 節 | 主題 | 図解 |
|---|---|---|
| 第1節 | OSI参照モデル | OSI 7階層の縦積み図 |
| 第2節 | カプセル化 | L4/L3/L2ヘッダが順に付く動く図 |
| 第3節 | 機器ごとの見ているヘッダ | PC、L2SW、ルータ、サーバの構成図 |
| 第4節 | MAC/IP/ポート番号 | 3層ヘッダ対応表 |
| 第5節 | ARP | 同一ネットワーク宛てと別ネットワーク宛てのARP比較図 |
| 第6節 | 通信の統合理解 | PCからWebサーバまでの動く図解 |
| 第7節 | VLAN | VLAN別ブロードキャストドメイン図、VLAN間通信図 |
| 第8節 | 午後問題の構成図読解 | L2境界、L3境界、ブロードキャスト範囲を色分けした構成図 |

図解がない節は、原則として実装未完了と扱う。

### 5.1 第1節 OSI参照モデルは通信を見るための地図

#### 目的

最初の節では、個別機器やMACアドレスではなく、通信を階層で見るための地図を作る。

#### 本文要点

- OSI参照モデルは丸暗記表ではなく、通信を分解して見るための枠組みである
- 上位層ほどアプリケーションに近く、下位層ほど物理的な伝送に近い
- ネスペでは、設問がどのレイヤの話をしているかを見分けることが重要
- 第1章では、まずL4、L3、L2、L1を中心に見る

#### 図解仕様

静的図: OSI 7階層の縦積み図。

表示内容:

- L7 アプリケーション層: HTTP、DNSなど
- L6 プレゼンテーション層: 文字コード、暗号化など
- L5 セッション層: セッション管理
- L4 トランスポート層: TCP/UDP、ポート番号
- L3 ネットワーク層: IP、ルーティング
- L2 データリンク層: Ethernet、MAC、L2SW、VLAN、ARP
- L1 物理層: ケーブル、光、電波、ビット

強調:

- 第1章の主対象としてL1〜L4をハイライト
- L7〜L5は「今後の章で深掘り」と薄く表示

#### NG

- L2から説明を始めない
- 「L2は同じLAN、L3は別ネットワーク」とだけ言って終わらせない

### 5.2 第2節 データには下位層へ渡すたびにヘッダが付加される

#### 目的

カプセル化を最初に理解させる。ここが分かると、フレーム、パケット、セグメントの違いが自然に入る。

#### 本文要点

- Webアクセスの出発点は、アプリケーションデータである
- L4でTCPヘッダが付き、TCPセグメントになる
- L3でIPヘッダが付き、IPパケットになる
- L2でEthernetヘッダとトレーラが付き、Ethernetフレームになる
- L1ではビット列として信号に変換される

#### 動く図解仕様

名称: カプセル化アニメーション

画面構成:

- 左: クライアントPC
- 中央: データの箱
- 右: OSIレイヤの縦スタック
- 下: 現在の説明文

ステップ:

1. アプリケーションデータを表示する
2. L4ヘッダが上から付く。表示名はTCPセグメント
3. L3ヘッダが外側に付く。表示名はIPパケット
4. L2ヘッダとL2トレーラが外側に付く。表示名はEthernetフレーム
5. L1でビット列としてケーブルへ出る

各ステップで強調するヘッダ:

- L4: 送信元ポート、宛先ポート
- L3: 送信元IP、宛先IP
- L2: 送信元MAC、宛先MAC、FCS

#### NG

- L4、L3、L2を横並びの知識として説明しない
- ヘッダが「同時に何となく付く」ような図にしない

### 5.3 第3節 クライアント、ネットワーク、サーバは同じデータを違う見方で扱う

#### 目的

エンドホストと中継機器の違いを理解させる。

#### 本文要点

- クライアントはアプリケーションデータを作り、下位層へ渡して包む
- L2SWはL2ヘッダのMACアドレスを見て転送する
- ルータは受信フレームのL2を外し、IPヘッダを見て経路判断する
- ルータは次のリンクへ出すために、新しいL2ヘッダを付ける
- サーバはL2、L3、L4を順に外し、アプリケーションデータを受け取る

#### 図解仕様

静的図: クライアント、L2SW、ルータ、Webサーバの構成図。

各機器の上に「見るヘッダ」を表示する。

- PC: L7/L4/L3/L2を作る
- L2SW: L2ヘッダを見る
- ルータ: L2を外してL3ヘッダを見る
- Webサーバ: L2/L3/L4を外してアプリへ渡す

#### 動く図解仕様

第2節のカプセル化アニメーションと連動させる。

ステップ:

1. PCでEthernetフレームが完成する
2. L2SWへ移動する。L2ヘッダの宛先MACをハイライト
3. ルータへ移動する。L2ヘッダを外すアニメーション
4. IPヘッダの宛先IPをハイライト
5. 次区間用のL2ヘッダを付けるアニメーション
6. Webサーバへ移動する
7. サーバでL2、L3、L4を外し、アプリケーションデータへ戻る

### 5.4 第4節 MACアドレス、IPアドレス、ポート番号の役割

#### 目的

MAC、IP、ポート番号を、カプセル化されたデータの中の位置と対応付けて理解させる。

#### 本文要点

- ポート番号は、エンドホスト内のアプリケーション通信を識別する
- IPアドレスは、最終的な通信相手を識別する
- MACアドレスは、同じリンク上の次の相手を識別する
- MACは区間ごとに変わる
- IPは基本的にエンドツーエンドで維持される。ただしNATなどの例外は後続章で扱う

#### 図解仕様

静的図: 3層ヘッダ対応表。

列:

- レイヤ
- データ単位
- 主な識別子
- 見る機器
- 見ている範囲

行:

- L4 / TCPセグメント / ポート番号 / エンドホスト / アプリケーション
- L3 / IPパケット / IPアドレス / ルータ / エンドツーエンド
- L2 / Ethernetフレーム / MACアドレス / L2SW / 同一リンク、同一VLAN

### 5.5 第5節 ARPはL2フレームを作るために必要になる

#### 目的

ARPを、カプセル化の流れの中で自然に登場させる。

#### 本文要点

- IPパケットを作れても、Ethernetフレームを作るには宛先MACが必要
- 同一ネットワーク宛てなら、宛先ホストのMACアドレスをARPで調べる
- 別ネットワーク宛てなら、デフォルトゲートウェイのMACアドレスをARPで調べる
- ARP要求はブロードキャスト
- ARP応答で得た対応をARPテーブルに保存する

#### 図解仕様

静的図: 同一ネットワーク宛てと別ネットワーク宛ての比較図。

左:

- PC A -> PC B
- ARPでPC BのMACを調べる

右:

- PC -> Webサーバ
- Webサーバは別ネットワーク
- ARPでデフォルトゲートウェイのMACを調べる

#### NG

- ARPを章の冒頭に置かない
- ARPを「IPアドレスからMACアドレスを知る」とだけ説明しない
- 何のためにMACアドレスが必要なのかを省略しない

### 5.6 第6節 動く図解: PCからWebサーバへ届くまで

#### 目的

第1節〜第5節で学んだ内容を、1つの動く図で統合する。

現行の動く図解の方向性は残す。ただし、単なるパケット移動ではなく、カプセル化、デカプセル化、再カプセル化を同時に見せる。

#### 画面構成

- 上段: PC、L2SW、ルータ、Webサーバのネットワーク構成図
- 中段: 現在運ばれているデータ構造
- 右側: 現在注目しているヘッダ
- 下段: ステップ説明と操作ボタン

#### ステップ

1. PCでアプリケーションデータを作る
2. L4ヘッダを付けてTCPセグメントにする
3. L3ヘッダを付けてIPパケットにする
4. ARPでデフォルトゲートウェイのMACアドレスを調べる
5. L2ヘッダ/トレーラを付けてEthernetフレームにする
6. L2SWが宛先MACを見てルータ側へ転送する
7. ルータがL2を外し、宛先IPを見て経路判断する
8. ルータが次区間用のL2ヘッダ/トレーラを付け直す
9. WebサーバがL2、L3、L4を外し、アプリケーションデータを受け取る

#### この図で必ず見せること

- L2フレームはリンクごとに作られる
- IPパケットはフレームの中に入っている
- ルータはIPパケットをむき出しで送るわけではない
- L4ヘッダはエンドホスト間の情報であり、ルータが作り直すものではない

### 5.7 第7節 VLANとブロードキャストドメイン

#### 目的

VLANは第1章の最後で扱う。L2の文脈ができた後に説明する。

#### 本文要点

- VLANはL2の範囲を論理的に分ける
- 同じ物理スイッチでも、VLANが違えばブロードキャストは届かない
- VLANを越える通信にはL3の中継が必要
- ARP要求がどこまで届くかを見ると、VLANの意味が分かりやすい

#### 図解仕様

静的図1: VLAN別ブロードキャストドメイン図。

表示内容:

- 1台のL2SW
- VLAN 10: PC-A、PC-B
- VLAN 20: PC-C、PC-D
- VLAN 10内だけに広がるARP要求の色付き範囲
- VLAN 20にはARP要求が届かないことを点線または薄い表示で示す

この図で伝えること:

- 同じ物理スイッチにつながっていても、VLANが違えば同じL2範囲ではない
- ブロードキャストは同じVLAN内に閉じる
- VLANは「配線の見た目」ではなく「L2の論理的な範囲」を分ける

静的図2: VLAN間通信図。

表示内容:

- VLAN 10のPC
- VLAN 20のサーバ
- L2SW
- L3SWまたはルータ
- VLAN 10内のL2フレーム
- L3SW/ルータでのL3中継
- VLAN 20側の新しいL2フレーム

この図で伝えること:

- VLANを越える通信は、L2だけでは届かない
- VLAN間通信にはL3の中継が必要
- VLAN間通信でも、各リンクではL2フレームとして運ばれる

#### 動く図解候補

第1章の初期実装では静的図を優先する。余力があれば、ARP要求がVLAN 10だけに広がり、VLAN 20には届かない様子を軽いアニメーションで見せる。

### 5.8 第8節 ネスペ午後では構成図の境目を読む

#### 目的

第1章の学習内容を試験対策に接続する。

#### 本文要点

- 構成図では、L2の境目、L3の境目、ブロードキャストドメインを読む
- 問題文の「同一セグメント」「デフォルトゲートウェイ」「ARP」「VLAN」「L3SW」はレイヤ判断のヒント
- 設問で理由を書くときは、どの機器がどのヘッダを見ているかを根拠にする

#### 図解仕様

静的図: 午後問題風の構成図読解マップ。

表示内容:

- PC、L2SW、L3SW、ルータ、サーバを含む小さな構成図
- VLAN 10、VLAN 20、サーバセグメントを色分け
- L2境界を緑、L3境界を青で表示
- デフォルトゲートウェイの位置を強調
- ARP要求が届く範囲を半透明の塗りで表示
- ルーティングが発生する点を矢印で表示

この図で伝えること:

- 構成図では、まず同じL2範囲を探す
- 次にL3中継が発生する場所を見る
- 設問の根拠は、機器名ではなく「どの範囲で、どのヘッダを見ているか」から探す

#### 演習的な見せ方

図の下に「この構成図で最初に見る場所」を3つ並べる。

1. 同じVLANかどうか
2. デフォルトゲートウェイはどこか
3. ARPが届く範囲はどこまでか

## 6. 第1章 実装方針

### 6.1 現行実装の扱い

現行の第1章1〜5節は、教材導線として採用しない。既存本文を部分修正して使い回さず、この詳細設計に基づいて作り直す。

第6節の動く図解は、方向性は残す。ただし、カプセル化、デカプセル化、再カプセル化を見せる図へ拡張する。

### 6.2 必要な実装変更

- `src/data/textbookChapters.ts` の第1章本文を全面置換する
- OSI参照モデル図を追加する
- カプセル化アニメーション用の新しい図解コンポーネントを追加する
- 既存の動く図解を、データ構造表示とヘッダ注目表示を持つ構成へ拡張する
- ARPは第5節へ移動する
- VLANは第7節へ移動する

### 6.3 既存ページへの影響

教科書モード以外には影響させない。ホームとサイドバーの入口は既存のまま維持する。

## 7. 第1章セルフレビュー観点

実装後、次を必ず確認する。

- 第1節がOSI参照モデルから始まっている
- 第1節でL2やARPが主題になっていない
- 各リンクでL2フレームとして運ばれる説明になっている
- ルータでL2を外し、IPを見て、次リンク用L2を作る説明になっている
- L4ヘッダをルータが付け直すような説明がない
- ARPが「L2フレームを作るために必要」という文脈で出ている
- 動く図解で、L4、L3、L2のヘッダが順に付く様子が見える
- 図だけを見ても、本文だけを読んでも、同じ理解に到達できる
- ネスペ午後で使える言い方になっている
- 実務の障害切り分けに使える見方になっている

### 9.8 第1節 L4はアプリケーション通信の入口

#### 目的

第1章で学んだOSI参照モデルの中で、L4が何を担当するかを明確にする。

初学者は「IPアドレスで宛先端末へ届く」までは理解しやすいが、届いたデータが端末上のどのアプリケーションへ渡されるかを見落としやすい。第1節では、L4を「端末内のアプリケーション通信を識別する層」として位置付ける。

#### 本文要点

- L3のIPアドレスは端末やインタフェースを識別する
- しかし、1台の端末ではWeb、DNS、メール、リモート接続など複数の通信が同時に動く
- そこでL4のポート番号を使い、どのサービスやアプリケーションの通信かを識別する
- L4の代表がTCPとUDPである
- TCP/UDPは、IPパケットの中に入るL4ヘッダとして見る

#### 図解仕様

静的図: OSI図とPC内部アプリ/ポート対応図。

表示内容:

- 左側にOSI 7層。L4を強調表示
- 中央にPC。ブラウザ、DNSクライアント、メールクライアント、管理ツールなどを小さな枠で表示
- 右側にポート番号例
  - HTTPS: TCP/443
  - DNS: UDP/53、TCP/53
  - SMTP: TCP/25
  - SSH: TCP/22
- 下に「IPアドレスは端末へ、ポート番号は端末内のサービスへ」という対比を表示

#### この図で伝えること

- L4はL3の代わりではない
- IPアドレスだけでは、端末内のどのアプリケーションへ渡すかは決まらない
- ポート番号は、後続章でFWルール、プロキシ、LB、NAPTを読むときの基礎になる

#### NG

- 「ポート番号はアプリケーションそのものを表す」と断定しない。正しくは、通信を渡すサービスやプロセスを識別する番号である
- 「L4は暗号化をする層」と説明しない。TLSは後続章で扱う

### 9.9 第2節 TCPとUDPは何が違うのか

#### 目的

TCPとUDPを、単なる「信頼性あり/なし」の暗記にしない。通信開始、順序制御、再送、オーバヘッド、用途の違いとして整理する。

#### 本文要点

- TCPは通信前にコネクションを確立する
- TCPは順序制御、確認応答、再送制御、フロー制御を持つ
- UDPはコネクション確立を行わず、データグラム単位で送る
- UDPは軽量だが、必要な信頼性はアプリケーション側で補うことがある
- DNS、DHCP、VoIP、CoAPなどはUDPを使う場面が多い
- HTTP/1.1、HTTP/2は基本的にTCP上で動く。HTTP/3はUDP上のQUICを使うが、この章では深入りしない

#### 図解仕様

静的図: TCP/UDP比較図。

表示内容:

| 項目 | TCP | UDP |
|---|---|---|
| 通信開始 | 3ウェイハンドシェイクでコネクション確立 | コネクション確立なし |
| 順序制御 | あり | なし |
| 再送制御 | あり | プロトコル自体にはなし |
| オーバヘッド | 比較的大きい | 小さい |
| 代表例 | HTTP/HTTPS、SMTP、SSH | DNS、DHCP、VoIP、CoAP |
| 午後問題で見る点 | 接続確立、セッション、遅延、LB/FW状態 | 保持時間、再送設計、軽量性 |

補助図:

- 同じIPパケットの中にTCPヘッダまたはUDPヘッダが入る比較
- TCPは矢印が往復してからデータ送信
- UDPはすぐデータ送信

- 「UDPは信頼性がないから重要な通信に使わない」と書かない

- HTTP/3/QUICを第2章の主題にしない

### 9.10 第3節 ポート番号をどう読むか

#### 目的

宛先ポートと送信元ポートの違いを整理する。初学者は「HTTPSは443」と覚えていても、通信の戻りがどのように元のアプリへ返るかを理解していないことが多い。

#### 本文要点

- 宛先ポートは、宛先端末上のどのサービスへ届けるかを示す
- 送信元ポートは、送信元端末上のどの通信へ戻すかを識別するために使われる
- クライアント側の送信元ポートは、多くの場合エフェメラルポートになる
- サーバ側の宛先ポートは、ウェルノウンポートや登録済みポートであることが多い
- 戻り通信では、送信元ポートと宛先ポートが入れ替わる
- FW、NAPT、LBは、このポート番号を通信識別に使う

#### 図解仕様

静的図1: 宛先ポートと送信元ポートの通信方向図。

表示内容:

- PC `192.168.10.10:51524`
- Webサーバ `203.0.113.20:443`
- 行き: `送信元 192.168.10.10:51524 -> 宛先 203.0.113.20:443`
- 戻り: `送信元 203.0.113.20:443 -> 宛先 192.168.10.10:51524`
- 送信元/宛先の入れ替わりを色で示す

静的図2: ポート番号分類表。

表示内容:

|---|---|---|---|
| ウェルノウンポート | 0-1023 | 代表的なサービス | 22, 25, 53, 80, 443 |
| 登録済みポート | 1024-49151 | 製品やアプリケーションが登録して使う | 例は本文で最小限 |
| 動的/プライベートポート | 49152-65535 | クライアント側の一時ポートなど | 51524など |

### 9.11 第4節 TCP 3ウェイハンドシェイク

#### 目的

TCPがデータ送信前にコネクションを確立することを、動く図で理解させる。

#### 本文要点

- TCPでは、データ送信前にSYN、SYN/ACK、ACKの3ステップでコネクションを確立する
- この時点では、HTTPリクエスト本文はまだ送っていない
- 3ウェイハンドシェイクは、双方が通信可能であることと、初期シーケンス番号を確認するための手順である
- FWやLBは、TCPの状態を見て通信を追跡する場合がある

#### 動く図解仕様

動く図: TCP 3ウェイハンドシェイク。

画面構成:

- 左: PC
- 中央: FWまたはL3SWを薄く表示
- 右: Webサーバ
- 下: 現在のL4ヘッダ情報
- 右下: このステップで見るポイント

ステップ:

1. PCからWebサーバへSYN
   - 送信元: `192.168.10.10:51524`
   - 宛先: `203.0.113.20:443`
   - TCPフラグ: SYN
   - 説明: PCがWebサーバのHTTPSサービスへTCP接続を開始する
2. WebサーバからPCへSYN/ACK
   - 送信元: `203.0.113.20:443`
   - 宛先: `192.168.10.10:51524`
   - TCPフラグ: SYN, ACK
   - 説明: Webサーバが接続要求を受け取り、応答する
3. PCからWebサーバへACK
   - 送信元: `192.168.10.10:51524`
   - 宛先: `203.0.113.20:443`
   - TCPフラグ: ACK
   - 説明: PCが応答を確認し、TCPコネクションが確立する
4. アプリケーションデータ送信の入口
   - TCPコネクション確立後に、TLSやHTTPの処理へ進むことを示す

#### この図で伝えること

- TCP接続確立とHTTPリクエスト送信は別の段階である
- 宛先ポート443は最初のSYNから使われる
- 送信元ポートは戻り通信でも重要である

### 9.12 第5節 TCP通信中に見ている状態

#### 目的

TCPを3ウェイハンドシェイクだけで終わらせず、通信中の順序制御、確認応答、再送、ウィンドウ制御の入口を説明する。

午後問題では、TCPの細かい計算よりも、処理待ち、遅延、並列性、接続状態、再送のような現象理解が必要になる。

#### 本文要点

- TCPはシーケンス番号でデータの順序を管理する
- 受信側はACKでどこまで受け取ったかを知らせる
- パケット損失時には再送が発生する
- ウィンドウ制御により、受信側が処理できる量を考慮する
- HTTP/2など上位プロトコルの改善は、TCPの上でどのように多重化するかという話につながる

- 右側に、シーケンス番号、ACK番号、ウィンドウサイズの簡略説明

### 9.13 第6節 UDPをどう読むか

#### 目的

UDPを「TCPの簡易版」として雑に扱わず、コネクションを持たない通信として読む。

#### 本文要点

- UDPはコネクション確立を行わない
- 1つのUDPデータグラム単位で送る
- DNS、DHCP、VoIP、CoAPなどで使われる
- 再送や順序制御が必要な場合は、アプリケーション側で補う
- FWやNAPTは、UDPにも一定時間の状態を持つことがある
- UDP保持時間が長すぎると、未使用フローのエントリーが残り続けることがある

#### 図解仕様

静的図1: UDP通信の図。

表示内容:

- PCからDNSサーバへUDP/53で問い合わせる簡略図
- 応答が戻る
- TCPのようなSYN/SYN-ACK/ACKがないことを明示

静的図2: UDP保持時間の比較図。

表示内容:

- NAPT/FWがUDPフローを一定時間保持する
- 保持時間が長い場合、未使用エントリーが残る
- 保持時間を短くすると、未使用エントリーを早く削除できる

#### 午後問題への接続

- R7 午後I 問2のUDP保持時間
- R7 午後II 問2のCoAP
- DNS、DHCP、VoIPの後続章

#### NG

- UDPには状態が一切関係ない、と書かない。UDP自体はコネクションレスだが、FW/NAPTなど中継装置が状態を管理することはある
- UDPは常に再送しない、と書かない。UDP自体に再送制御はないが、アプリケーションが再送することはある

### 9.14 第7節 5タプルで通信を識別する

#### 目的

午後問題で頻出するFW、NAPT、LB、プロキシ、セッション維持へつなげるため、通信を5タプルで識別する考え方を作る。

#### 本文要点

- 通信は、送信元IP、宛先IP、プロトコル、送信元ポート、宛先ポートの組み合わせで識別できる
- これを5タプルと呼ぶ
- 同じPCから同じWebサーバへ複数接続しても、送信元ポートが違えば別通信として識別できる
- FWは5タプルやTCP状態を使って通信を追跡する
- NAPTはIPアドレスとポート番号を書き換えて、多数の内部端末を外部通信へ対応付ける
- LBはセッション維持や振り分けでL4情報を利用することがある

#### 図解仕様

| プロトコル | TCP | TCPかUDPか |
| 送信元ポート | 51524 | 送信元側の一時ポート |
| 宛先ポート | 443 | 宛先側のサービス |

### 9.16 第2章 実装方針

#### 9.16.1 データ配置

第2章は、`src/data/textbookChapters.ts` のdraft章 `layer4-7` を置き換える形で実装する。

実装時の章IDは既存URLとの整合性を考え、まずは `layer4-7` を維持する。ただし、表示タイトルは「TCP/UDPとポート番号: アプリケーション通信の入口を読む」に変更する。

#### 9.16.2 既存図解型の利用

利用候補:

- `layer-stack`: 第1節のOSI内L4強調
- `comparison`: 第2節のTCP/UDP比較
- `address-role-table`: 第3節のポート番号整理表に転用可能
- `sequence`: 第4節の3ウェイハンドシェイクの静的補助
- `packet-frame`: 第2節、第7節のヘッダ/5タプル表現
- `interactive-flow`: 第4節の3ウェイハンドシェイク
- `exam-network`: 第8節の午後問題風観測点図

#### 9.16.3 必要になり得る追加図解型

第2章実装時に、既存型で見づらい場合は次の追加を検討する。

- `port-role-table`: 宛先ポート、送信元ポート、ウェルノウンポート、エフェメラルポートを表で見せる専用型
- `flow-identity-table`: 5タプルとFW/NAPT状態表を横に並べる専用型
- `tcp-handshake-flow`: SYN、SYN/ACK、ACKとTCPフラグを見せる専用動く図

ただし、まず既存型で実装可能かを確認する。第2章だけのために過剰な抽象化を増やさない。

### 9.17 第2章 セルフレビュー観点

実装後、次を必ず確認する。

- 第2章がHTTPやDNSの詳細へ逸れていない
- L4の役割が「端末内のアプリケーション通信を識別すること」として説明されている
- 宛先ポートと送信元ポートの違いが図と本文で分かる
- 戻り通信でポート番号が入れ替わることが示されている
- TCP 3ウェイハンドシェイクでHTTPリクエストを同時に送っているような誤解がない
- UDPを「信頼性がないから使えない」と説明していない
- UDP保持時間とNAPT/FW状態管理の関係が、UDP自体のコネクションレス性と矛盾していない
- 5タプルが暗記項目ではなく、FW、NAPT、LB、ログ読解への入口として説明されている
- 図解が全節にある
- 動く図解のラベルやボタンが文字被りしていない
- モバイル幅で、ポート番号表と5タプル表が横にはみ出しすぎない
- 第3章以降への接続が自然に書かれている
- 日本語が不自然な翻訳調になっていない
