実装済みの主な内容:

- 全20章の「読解の足場」図
- スマホ、タブレット、PC幅の主要表示対応

現在の章状態:

- 全20章が `status: 'diagram'`
- 本文説明文は未完成
- 図解内ラベルと短い説明はあるが、教材本文としては不足
- 完成版の公開品質には達していない

直近で実施済みの技術修正:

- DNS/DHCP/ARPを直列経路ではなく、通信前準備・分岐トポロジとして扱うよう修正
- SSO/PKIで、IdPからアプリへ直接送るように見える誤解を避け、ブラウザ、SP、IdPのフロントチャネル中心の流れへ修正
- メール配送で、DNS参照とSMTP配送を分け、DNSがMXへ配送するような図にならないよう修正
- VoIPで、SIP制御とRTPメディア経路を分離
- IPv6 RAの方向を、ルータからホスト側へ修正
- HTTPメソッドやHTTPステータスをポート番号欄へ入れないよう修正
- 分岐トポロジでも、構成図、動く図解、読解の足場図が同じリンク定義を使うよう修正

## 4. 今後やるべきこと

最優先は、第1章から順番に「教材として読める本文」を作ることである。

推奨順序:

1. 第1章から第4章を完成版に近づける
2. 第1章から第4章で、本文のトーン、図解説明、スマホ表示、強調表現の型を固める
3. 第5章以降へ同じ品質基準を展開する
4. 章固有の専門図を追加する
5. 必要に応じて増補図解を追加する
6. 章ごとにセルフレビューして `status: 'published'` へ進める

各章の作業単位:

1. 設計書を読む
2. 既存図解を技術レビューする
3. 既存図解が不十分なら修正または作り直す
4. 図の前後に自然な本文説明を追加する
5. 重要語を太字、色、表で整理する
6. 初学者が先に知っていない用語を唐突に出していないか確認する
7. ネスペ午後問題の読解行動へ接続する
8. PC、タブレット、スマホで表示確認する
9. build / lint / ブラウザ確認を行う
10. 実装計画書へメモを残す
11. commit / push する

## 5. 裁量

Claude Codeには、次の裁量を渡す。

- 教科書モード内に閉じる限り、本文、図解データ、図解コンポーネント、章内構成を修正してよい
- 技術的に誤っている図、読みにくい図、初学者に不親切な図は作り直してよい
- 過去の実装に引きずられず、詳細設計に合わせて再設計してよい
- 図が混みすぎる場合は、構成図、ヘッダ図、シーケンス図、観測点図、変更前後図へ分割してよい
- 章固有の専門図、ログ風表、設定表、パケットキャプチャ風表を追加してよい
- 技術判断に不安がある場合は、IPA公式資料、RFC、主要ベンダ資料など一次情報を確認してよい
- 2章以上で同じ図解型が安定した場合は、共通コンポーネント化してよい

ただし、次は不可とする。

- ノートモード、クイズ、午後問題演習、同期、XP、勲章、活動履歴との連携を追加する
- ホームとサイドバーの入口導線以外の既存画面へ影響を出す
- 画像ファイルを追加する
- 図解だけを増やして、本文で意味を説明しない
- 技術的に曖昧なまま推測で図や説明を確定する
- スマホで横スクロールしないと読めないネットワーク構成図を提出する
- 線、ラベル、バッジ、機器名が重なった状態で提出する

## 6. 品質基準

顧客提出前に、必ず次の観点でレビューする。

ユーザ目線:

- 初学者が、図を見た瞬間にどこを見るか分かる
- 本文の日本語が自然で、読み物として読める
- 重要語、混同しやすい語、試験で問われる判断材料が強調されている
- 図の直後に、その図で何を見るかが説明されている

ネットワークエンジニア目線:

- L2、L3、L4、L7の役割を混同していない
- 各リンク上ではL2フレームとして運ばれ、その中にIPパケットやTCP/UDPセグメントが入るという前提を崩していない
- ルータ、L3SW、FW、LB、プロキシ、DNS、認証基盤、監視基盤が見る情報を混同していない
- 物理構成、論理構成、制御プレーン、データプレーンを混同していない
- 実務で障害切分けや設計レビューに使える観測点がある

IPA試験作成者目線:

- 午後問題の構成図、条件、設問、解答根拠を読む訓練になっている
- 宛先判断、経路判断、許可条件、障害原因、変更影響、答案化へつながっている
- 単なる用語説明で終わっていない
- 「なぜ」「どこ」「何が変わるか」を説明できる

## 9. 最初に確認すべきリスク

- 第1章から第4章の本文が未完成のため、読者が学習順を掴めない
- 図解が全章にある一方、章固有の深掘り図が不足している
- 共通図解コンポーネントの変更が全章へ波及する
- スマホ表示は全章確認済みだが、本文追加後に余白、改行、ボタン位置が崩れる可能性がある
- 技術的に細かい章では、既存の短いラベルだけでは誤解を防ぎ切れない

## 10. 引き継ぎ後の最初の完了条件

最初の完了条件は、第1章を「本文付きの完成候補」にすることである。

第1章の完了条件:

- OSI 7階層、カプセル化、デカプセル化、再カプセル化を自然な順序で説明している
- MACアドレス、IPアドレス、ポート番号の役割と表記を表で整理している
- 同一ネットワーク宛てと別ネットワーク宛ての違いを、ARPと次ホップの観点で説明している
- 図解がスマホでもネットワーク構成図として読める
- 動く図解で、L4、L3、L2ヘッダが付く様子と、ルータでL2だけ付け替わる様子が分かる
- 本文の日本語が自然で、初学者が読み続けたくなる
- `npm run build` と `npm run lint` が成功する
- 実装計画書にメモを追記し、commit / push が完了している
