# 図解で学ぶ教科書 図解詳細設計マトリクス

作成日: 2026-06-14

## 1. 目的

この文書は、各章の各節に置く図解を、実装前に迷わない粒度へ落とすための補助設計書である。

章別詳細設計書の節構成表は「何を扱うか」を定義する。本書では、それを受けて、各節の図解について次を固定する。

- 何を説明するための図か
- どういう形の図にするか
- 動く図解の場合、何をどう動かすか
- 読者に何を見てほしいか
- 午後問題のどの読解行動に効くか

この文書の該当行を確認せずに本文実装へ進まない。

## 2. 図解レビューの共通基準

### 2.1 ユーザ目線

- 初学者が、図を見た瞬間に「どこを見ればよいか」が分かる。
- 文章だけでは想像しづらい範囲、順番、違いが図で見える。
- 黒一色の細かい図にせず、重要な線、境界、ヘッダ、状態を控えめに強調する。
- 図の直後に「この図で見るところ」を本文で説明する。

### 2.2 ネットワークエンジニア目線

- 機器が見ているヘッダ、転送判断、状態管理、ログ取得点が技術的に正しい。
- L2、L3、L4、L7の役割を混同しない。
- 物理構成と論理構成、制御プレーンとデータプレーンを混同しない。
- 図が実務の障害切分け、設計レビュー、変更影響確認に使える。

### 2.3 IPA試験作成者目線

- 午後問題の構成図、条件、設問、解答根拠を読む訓練になる。
- 単なる用語説明ではなく、設問で問う「なぜ」「どこ」「何が変わるか」に接続する。
- 採用理由、採用しない場合の問題、トレードオフを説明できる。
- 年度・問題名の列挙だけでなく、読ませる境界、経路、条件、観測点が明確である。

## 3. 図解詳細設計フォーマット

| 項目 | 内容 |
|---|---|
| 説明目的 | この図で理解させること |
| 図の型 | 構成図、シーケンス図、ヘッダ図、比較表、状態遷移図、観測点図 |
| 構成要素 | 機器、境界、サブネット、VLAN、ヘッダ、ログ、状態、設定 |
| 強調対象 | 最初に見る線、境界、ヘッダ、状態、表の列 |
| 動き | 動く図解の場合、ステップごとに動くものと変わる説明 |
| 読解行動 | 午後問題での宛先判断、経路判断、許可条件、障害原因、変更影響、答案化 |
| 表示品質 | 文字被り、線被り、モバイル幅、ボタン位置の確認 |

## 3.5 図解を後から増やすための設計余地

本書の章別・節別マトリクスは、各節に最低限必要な図解を定義するものである。実装後のレビュー、読者要望、過去問分析の追加結果によって、図解を増やす余地を最初から残しておく。

図解は次の2種類に分けて管理する。

| 種別 | 役割 | 実装判断 |
|---|---|---|
| 基本図解 | その節の理解に必須となる図。各節に原則1つ以上置く | 図解先行実装フェーズで作る |
| 増補図解 | 読者がつまずきやすい点、実務で効く観測点、午後問題の追加読解に対応する図 | 基本図解のレビュー後、必要度が高いものから追加する |

増補図解は、単に図の枚数を増やすために追加しない。次のいずれかに該当する場合に追加する。

- 基本図解だけでは、初学者が「どこを見るか」を誤解しやすい
- 同じ構成図で、正常時、障害時、変更後を比較した方が理解しやすい
- パケットキャプチャ、ログ、設定表など、午後問題で別資料として出やすい情報を重ねて見せたい
- 実務では必ず確認する観測点があるが、基本図解に入れると図が混みすぎる
- L2、L3、L4、L7、制御プレーン、データプレーンの混同を防ぐために、別視点の図が必要である

増補図解を追加する場合は、既存図に無理に詰め込まない。図が読みにくくなる場合は、同じ題材を「構成図」「ヘッダ図」「観測点図」「変更前後図」に分ける。

## 5. 設計判断・トレードオフの入れ方

各章の本文には、少なくとも1回は「なぜその方式を使うのか」「使わない場合に何が困るのか」を入れる。

| 観点 | 本文に入れる例 |
|---|---|
| コスト | 専用線、冗長化、クラウド利用、運用負荷の違い |
| 信頼性 | 単一障害点、収束時間、状態同期、監視可能性 |
| 安全性 | 認証、暗号化、境界分離、ログ取得、証明書信頼 |
| 性能 | 遅延、帯域、キャッシュ、HTTP/2、QoS |
| 運用性 | 切戻し、設定変更範囲、監視、障害切分け |

午後問題での答案化に使うため、本文では「便利だから」「安全だから」で止めず、対象、理由、効果を短く書く。

例:

- FWの許可先をSaaS全体ではなくSWGへ集約することで、許可条件とログ取得点を一元化できる。
- DNS TTLを短くすると、切替後のIPアドレスが反映されるまでの時間を短縮できるが、問い合わせ数は増えやすい。
- VRRPはデフォルトゲートウェイの冗長化に有効だが、全ての上位通信障害を自動検知するわけではない。
