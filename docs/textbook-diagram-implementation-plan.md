# 図解で学ぶ教科書 図解先行実装計画書

作成日: 2026-06-14

## 1. 目的

この計画書は、「図解で学ぶ教科書」の各章について、本文説明文より先に図解と動く図解を実装するための実装計画である。

今回の実装フェーズでは、説明文の完成を目的にしない。先に、各章で読者が見るべき構成図、ヘッダ図、シーケンス図、状態遷移図、パケットキャプチャ風の動く図を作る。

実装時は、この計画書に実装メモを追記し、その追記を含めて commit / push する。

## 2. 前提文書

図解の仕様は、次の設計書を正とする。

- `docs/textbook-curriculum-architecture.md`
- `docs/textbook-chapter-detailed-design.md`
- `docs/textbook-chapters-03-20-detailed-design.md`
- `docs/textbook-diagram-detail-matrix.md`

実装中に既存コードと設計書が矛盾する場合は、既存コードではなく設計書を優先する。特に、現行の `src/data/textbookChapters.ts` の第2章以降の章順は、全体アーキテクチャの20章体系に合わせて整理する。

## 3. 今回フェーズの範囲

### 3.1 実装するもの

- 第1章から第20章までの章ページを、図解先行の状態で整備する
- 各章に、最低1つ以上の動く図解を置く
- 原則として各節に図解を置く
- 図解作成済みの章は、ページ上部に「工事中」と表示する
- 図解内には、理解に必要なラベル、凡例、ステップ名、観測点、ヘッダ表示、短い補助文を置く
- 図解実装後は、この計画書の「実装メモ」に対象章、追加した図、検証結果、commit subject を記録してから push する

### 3.2 今回フェーズで実装しないもの

- 完成版の本文説明文
- 暗記用のまとめ本文
- 章末テスト、確認問題、学習進捗
- ノートモードや他コンテンツとの連携
- 外部画像ファイルの追加

ただし、図解を成立させるための最小限のラベル、ステップ説明、操作ボタン名、キャプチャ欄の項目名は実装対象に含める。

## 4. 実装方針

### 4.1 章状態

図解先行フェーズ用に、章の状態を次の3段階で扱う。

| 状態 | 意味 | ページ表示 |
|---|---|---|
| `draft` | 未実装。章一覧には準備中として出す | 本文ページは準備中 |
| `diagram` | 図解だけ先行実装済み。説明文は未完成 | 本文ページを表示し、上部に工事中を出す |
| `published` | 本文、図解、レビューが完了 | 通常表示 |

現行コードの `published` / `draft` だけでは、図解先行公開と本文完成を分けられない。最初の実装作業で `diagram` 相当の状態を追加する。

工事中表示の文言案:

> 工事中: この章は図解部分のみ先行公開中です。説明文は後続工程で追加します。

### 4.2 コンテンツ分離

教科書モードの変更は、次の範囲に閉じる。

- `src/pages/Textbook.tsx`
- `src/pages/TextbookChapter.tsx`
- `src/data/textbook*.ts`
- `src/components/textbook/*`
- `docs/textbook-*.md`

ホームとサイドバーの入口リンク以外には影響を出さない。既存のノートモード、午後問題演習、コラム、同期機能とは連携しない。

### 4.3 図解コンポーネント

既存の図解コンポーネントを再利用しつつ、足りない型は教科書モード内に追加する。

- 午後問題風ネットワーク構成図
- 動くシーケンス図
- 動くヘッダスタック図
- パケットキャプチャ風テーブル
- 物理構成と論理構成の比較図
- 状態遷移図
- 正常時/障害時/変更前後の比較図

図解はSVGまたはHTML/CSSで実装し、画像ファイルは追加しない。

### 4.4 動く図解の最低条件

動く図解は、点が線上を動くだけでは完了扱いにしない。各ステップで次を表示する。

- 今どの装置が処理しているか
- その装置がどのヘッダ、状態、設定、ログを見ているか
- 宛先、経路、許可、状態変化のどれを判断したか
- パケットキャプチャ風に見た場合、L2/L3/L4/L7のどこが変わるか

操作は、前へ、再生/停止、次へ、最初からを持たせる。操作ボタンは説明文量で上下に揺れない固定配置にする。

### 4.5 増補図解の扱い

各節に置く基本図解を先に実装し、その後のレビューや要望に応じて増補図解を追加する。増補候補は `docs/textbook-diagram-detail-matrix.md` の「増補図解候補」を参照する。

増補図解は、既存図へ情報を詰め込むためではなく、次の目的で追加する。

- 基本図解だけでは誤解が残る箇所を別視点で見せる
- 正常時、障害時、変更前後を比較する
- パケットキャプチャ、ログ、設定表など午後問題で出る別資料を重ねる
- 実務で見る観測点を補う

実装メモには、対象図解が「基本図解」か「増補図解」かを記録する。

## 6. 品質ゲート

### 6.1 技術正確性

- 各リンク上ではL2フレームとして運ばれ、その中にIPパケットやTCP/UDPセグメントが入ることを崩さない
- L2SW、ルータ、FW、LB、プロキシ、DNS、認証基盤が見る情報を混同しない
- 物理構成、論理構成、制御プレーン、データプレーンを図上で混ぜない
- VLAN、DMZ、NAT、FW、証明書、BGP、VRRPなどを必要な章でだけ登場させ、唐突に出さない

### 6.2 表示品質

- 線、矢印、ラベル、機器名が重ならない
- バッジや吹き出しで文字を隠さない
- 320px、768px、PC幅で破綻しない
- 図の操作ボタンは説明文量で位置が大きく動かない
- 黒一色の細かい図にせず、境界、経路、状態、ヘッダを控えめに色分けする

### 6.3 レビュー観点

実装単位ごとに、次の3者の観点でセルフレビューする。

- 初学者目線: どこを見ればよいか分かるか
- ネットワークエンジニア目線: ヘッダ、経路、状態、ログの扱いが正しいか
- IPA試験作成者目線: 構成図、条件、設問根拠を読む訓練になっているか

## 7. 検証手順

各実装単位で、少なくとも次を行う。

1. `npm run build`
2. 対象章をブラウザで確認
3. PC幅、タブレット幅、スマホ幅でスクリーンショット確認
4. 動く図解の前へ、再生/停止、次へ、最初からを確認
5. 線と文字の重なり、不要な線、ラベルの欠けを確認
6. 技術正確性、話の流れ、日本語ラベルをセルフレビュー

## 8. push と実装メモの運用

図解を実装したら、push 前に必ずこの計画書の「9. 実装メモ」へ行を追加する。

行には次を記録する。

- 日付
- 対象章
- 実装した図解
- 実装した動き
- 工事中表示の有無
- 検証結果
- commit subject
- 残課題

commit hash は、その commit 自身の中へ正確に書き込むことができないため、計画書には commit subject を記録し、最終報告で実際の commit hash と push 結果を報告する。
