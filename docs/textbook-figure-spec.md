# 教科書モード 図解仕様書 v1

作成日: 2026-06-23

「どの概念を、どの図で、どう見せれば一番わかりやすく・見やすいか」を確定する。図解は本モードの最重要・差別化要素なので、**図の選択と画面仕様をここで正とする**（章別設計 `textbook-chapter-designs.md` の図はこの仕様に従う）。

---

## 1. 図解の設計原則（分かりやすさ・見やすさ最優先）

1. **1図＝1理解単位**。1枚に2つ以上の主張を詰めない。詰まるなら図を分ける／ステップに分ける。
2. **図の形は概念に合わせて選ぶ**（§2 の使い分け表）。「何を主役にするか」で図種が決まる：
   - 経路（どこを通るか）→ 構成図を旅する図
   - 順序（誰が何をどの順で）→ 縦のシーケンス図
   - 構造（入れ子・内訳）→ 入れ子図／対応表
3. **動きには必ず役割**。動かすのは「順を追う」「変化を見せる（変わる/変わらない）」「前後比較」のため。装飾アニメ・無限アニメは禁止。
4. **スマホ最優先・1画面**。375/360で横スクロール0、図＋操作が概ね1画面。文字は最小12px、縮小に頼らない。
5. **操作ボタンは動かない**。説明文・表・ステージは固定高にし、ステップを送ってもボタンの位置が変わらない（第1章で確立）。
6. **避ける失敗（実績あり）**: 横スクロール／文字・線・ラベルの重なり／動く文字が線を隠す／死にスペース過多／ボタンが上下に動く。各図の Don't に明記する。

## 2. 図の使い分け（概念タイプ → 最適な図）

| 見せたいことの主役 | 最適な図 | 例 |
|---|---|---|
| 層の地図（いまどの層か） | `osi-stack` | OSI 7層 |
| ヘッダの入れ子・包む/外す | `encap` | カプセル化、IPsecトンネル |
| 数項目を同じ目線で対比 | `address-table` | MAC/IP/ポート、TCP/UDP、L4/L7、認証/認可 |
| パケットが**構成図を旅する**・付け替え | `packet-flow` | 第1章の通信フロー、NAT書換、FW通過/遮断、フェイルオーバー迂回 |
| **2〜3者のやり取りの順序** | `sequence`（新） | DHCP・TCP3way・TLS・HTTP・RADIUS |
| 規則／対応表（特定行を強調） | `record-table`（新） | 経路表、FWルール、NAT変換表、MAC表 |
| アドレスの内部構造 | `subnet-calc`（新） | サブネット（ネットワーク部/ホスト部） |
| 手順の時系列俯瞰 | `timeline`（新） | 第1部パノラマ、無停止更改、証明書チェーン |

> **重要な使い分け**: 「`packet-flow` か `sequence` か」。
> - **`packet-flow`** = 主役が**経路／トポロジ**。複数機器をまたいで進み、区間ごとに宛先MACが変わる・NATで書き換わる等の**変化**を見せる。
> - **`sequence`** = 主役が**会話の順序**。2〜3者の間で「SYN→SYN/ACK→ACK」のような往復を、時間を縦に流して見せる。ハンドシェイク・要求応答・認証はすべてこちら（packet-flow で1本の線上を往復させるより圧倒的に読みやすい）。

## 3. 図コンポーネント・カタログ（確定仕様）

各図: **目的 / いつ使う / 画面構成（モバイル基準→PC）/ 動き / データ schema / 制約適合 / Do・Don't**。

### 3.1 `osi-stack`（既存）
- 目的: 7層の地図。いまどの層の話かを指させる。
- 画面構成: 縦に7行、各行 = [層番号(左) | 層名＋例(右上) | 一言説明(右下)]。主役層(L1〜L4)は色、上位(L5〜L7)は淡色(dim)。全幅・縦積み。
- 動き: 基本静的。任意で主役層を順にハイライトするステップ。
- schema: 既存 `OsiStackFigure`。
- 制約適合: 7行で1画面ほぼ一杯。説明は各行内の1行に収める。
- Do: 主役を強調・上位をdim。 Don't: 7層を同じ濃さで詰める。

### 3.2 `encap`（既存）
- 目的: フレーム⊃パケット⊃セグメント⊃データ の入れ子と、包む/外すの対称。
- 画面構成: 同心の入れ子ボックスを縦展開（外=L2→内へ）。**ステージは固定高（最大＝フル）で中身中央**。送/受フェーズのバッジ。
- 動き: 「次へ」で1枚ずつ包む→受信で外す。ステージ固定高でボタン不動（第1章で確立）。
- schema: 既存 `EncapFigure` を**任意の入れ子レベル（ラベル・色を外から渡す）に一般化**する（現状は L2/L3/L4 固定）。VPN(第11章)=最内「元IPパケット」＋外側「新IP＋IPsec」、VXLAN(第16章)=最内「元フレーム」＋外側「VXLAN/UDP/IP」を、この一般化 levels で表す。
- Do: ステージ固定高。 Don't: 包む深さでボタンを上下させる。

### 3.3 `address-table`（既存・対比に汎用）
- 目的: 数項目を同じ軸で並べて違いを際立たせる。
- 画面構成: **モバイル=1カラム縦積みカード**（PC=2〜3カラム）。各カード = 見出し＋[項目:値]の定義リスト。
- 動き: 静的。
- schema: 既存 `AddressTableFigure`（rows を対象に応じて差し替え）。
- Do: 比較軸を全カードで揃える。 Don't: 横長の表（横スクロールの元）。

### 3.4 `packet-flow`（既存・拡張）
- 目的: パケットが**構成図の上を旅する**過程と、区間ごとのヘッダ変化・機器処理。
- いつ使う: 経路・ホップ・宛先MAC付け替え・NAT書換・FW通過/遮断・フェイルオーバー迂回。
- いつ使わない: 2者間のプロトコル往復（→`sequence`）。
- 画面構成（モバイル）: 上=トポロジ（縦リフロー・全幅ノード・現区間ハイライト＋パケット移動）／中=ヘッダ表（L2/L3/L4 を**各行固定高**＋「変わる/そのまま/外す」チップ）／下=説明（固定高）＋操作。**全要素固定高でボタン不動**。
- 動き: 「次へ」で区間/処理が進む。有限トランジション。`prefers-reduced-motion` 時は移動を抑えハイライトのみ。
- 拡張: `verdict?:'pass'|'block'`（FWの通過/遮断チップ）／ノード `down`（故障表示＋経路迂回）。
- schema: 既存 `PacketFlowFigure`＋上記2拡張。
- Do: 第1章の固定高方式を踏襲。 Don't: ラベルが線/ノードに被る、ボタンが動く。

### 3.5 ★`sequence`（新・シーケンス／ラダー図）— プロトコルの往復を最も明快に
- 目的: 2〜3者の「誰が・何を・どの順で」を、時間が下へ流れるラダーで見せる。
- なぜ最適: 往復は「経路」より「順序」が主題。縦の生命線＋順番の矢印が、最も標準的で直感的。1本の線上で packet を往復させるより誤解が少なく見やすい。
- 画面構成（モバイル）: 上端にアクター見出し（**最大3**: 例 `PC` | `Webサーバ`）。各アクターから縦の生命線。メッセージ＝生命線間の横矢印を**上から順に全部描画**し、**現在ステップだけ強調**（色＋太線、他は淡色）。矢印ラベル=メッセージ名（SYN／SYN・ACK…）、その下に補足1行。ブロードキャストは破線＋「全員あて」。
- 動き: 「次へ」で強調が次の矢印へ移り、説明が変わる。**全矢印を最初から描くので総高が一定＝ボタンが原理的に動かない**。
- schema: `{ kind:'sequence', actors:{id,label,sub?}[], messages:{from,to,label,note?,style?:'normal'|'broadcast'}[] } & FigureBase`
- 制約適合: アクターはモバイル最大3（列幅=幅÷アクター数）。横スクロール0。
- **DNSの分解（4者以上の扱い）**: スタブ/キャッシュ/ルート/TLD/権威の5者は、3カラム `PC｜キャッシュDNS｜上位DNS` に固定。上位DNSへの問い合わせを「ルート→TLD→権威」と**段階ラベル**で繰り返す（右カラムの相手名を毎ステップ差し替えるのではなく、ラベルで段階を示す）。
- Do: 矢印の向き・順序を主役に、現在だけ強調。 Don't: 列を増やしすぎる、ラベルを長文にする。
- 適用: DHCP DORA（PC｜DHCP）／TCP3way（PC｜Web）／TLS（PC｜Web）／HTTP要求応答（PC｜Web）／RADIUS（利用者｜認証装置｜RADIUS）。

### 3.6 ★`record-table`（新・規則/対応表）
- 目的: 行＝1レコードの規則・対応を並べ、必要に応じ**特定行を強調**。
- 画面構成（モバイル）: **1行=1カードに縦積み**（横長表は禁止）。カード内は「列ラベル:値」を縦並び。`highlightRow` で該当行（ロンゲストマッチの一致行・適用FWルール等）を色で強調。PC=表形式も可。
- 動き: 静的、または隣の `packet-flow` と連動して「この通信→この行」と強調行を切替。
- schema: `{ kind:'record-table', columns:{key,label}[], rows:Record<string,string>[], highlightRow?:number } & FigureBase`
- 制約適合: モバイルは列3〜4に絞る。横スクロール0。
- Do: 強調で「いまどの行か」を明確に。 Don't: 横スクロールする表。

### 3.7 ★`subnet-calc`（新・アドレス構造）
- 目的: IPの「ネットワーク部／ホスト部」の境界を一目で。
- 画面構成（モバイル）: 4オクテットを縦2段——上段=10進(192/168/10/10)、下段=2進(8ビット×4、等幅)。プレフィックス長の位置に**縦の境界線**、左(ネットワーク部)=緑／右(ホスト部)=グレー。下にネットワークアドレス／ブロードキャスト／ホスト範囲を1行ずつ。
- 動き: 任意で「/24→/26」スライドし境界が右へ動く＋使えるホスト数が変わるステップ。
- schema: `{ kind:'subnet-calc', ip:string, prefix:number, show?:('range'|'broadcast')[] } & FigureBase`
- 制約適合: 32ビットは**オクテット改行**で4行に（横1列に並べない）。
- Do: ビット列は等幅・色分け。 Don't: 32桁を1行横並び（潰れる）。

### 3.8 ★`timeline`（新・手順の俯瞰）
- 目的: 一連の手順を時系列で俯瞰。
- 画面構成（モバイル）: 縦タイムライン。各項目 = [段階/アクターのバッジ | 何を(短文)]。
- 動き: 任意で現在項目ハイライトのステップ式。全項目描画で総高一定＝ボタン不動。
- schema: `{ kind:'timeline', items:{badge:string,label:string,detail?:string}[] } & FigureBase`
- 適用: 第1部パノラマ（DHCP→DNS→ARP→TCP→TLS→HTTP）／無停止更改の段取り／証明書チェーン（ルートCA→中間→サーバ）。
- Do: 1項目1行で短く。 Don't: 各項目に長文。

### 3.9 ★領域フォーカス（複雑トポロジのスマホ1画面化モード）
- 目的: 第8章以降の大きな構成図を、横スクロール0のまま「全体も現在地も」見せる。`TopologyView` のモードとして実装し `packet-flow`/`topology` が使う。
- 画面構成（モバイル）: 上=**俯瞰**（ゾーンを四角で並べた縮小図。ノードは描かずゾーン名・境界だけ、現在ゾーンを強調。高さ固定 約80px）／下=**詳細**（現在ゾーン内のノードだけを第1章と同じ全幅カードで表示）。パケット/ハイライトは詳細側で動く。
- 動き: ステップが別ゾーンへ移ると、俯瞰の強調が移り詳細が次ゾーンへ差し替わる（有限）。俯瞰は常時表示で「全体のどこか」を見失わせない。
- データ: 既存 `Topology`（zones/nodes/links）で足りる。`focus` のノード/リンクが属するゾーンを「現在ゾーン」とし、俯瞰=ゾーン一覧、詳細=そのゾーンのノード。**新フィールド不要**。
- 切替: ゾーン3つ以上の章（目安 第8章以降）で自動適用。少ない章（第1〜7章）は従来の全ノード縦リフロー。
- 制約適合: 詳細側は第1章の全幅カード方式（固定高・ボタン不動）を踏襲。横スクロール0。
- Do: 俯瞰は「ゾーンの地図」に徹し詰め込まない。 Don't: 全ノードを1画面に並べて潰す。
- 適用: 第8〜19章のトポロジ、第20章キャプストーン（午後相当の全体図）。

## 5. 実装メモ（図コンポーネント）

- 既存 `FigureRenderer` の switch に新 kind（`sequence`/`record-table`/`subnet-calc`/`timeline`）を追加し、`types.ts` の `Figure` union と各 schema を追加。
- 新図も**第1章の検証フロー**を必ず通す: 375/360で横スクロール0／全ステップ送り／操作ボタンの top 固定／`tsc` 0／コンソール無警告。
- `sequence`・`timeline` は「全要素を描画し現在だけ強調」する構造なので、**設計上ボタンが動かない**（固定高の作り込みが最小で済む）。これも図種選択の利点。
- まず第2章で `sequence`（DHCP/DNS）と `timeline` を実装して**型を確立**し、以降の章で再利用する。

## 6. 追補（第7・8章実装〜第1〜8章レビューで確定した拡張・規約）

- **stack N段**（GraphTopology）: spine（nodes配列順=上→下）を縦一列に。最上段の葉=上の横並び・最下段=下・中間段=左右の短い水平枝（幅96・中心±110・最大2つ）。最上段に葉が無ければ上の余白を詰める。2段時は従来と同一描画（第7章回帰なし）。
- **`Topology.leafIds`**: ロールに関わらず葉として扱うノード指定（例: 第8章BGP図で境界ルータ(router)をISPの雲(cloud)の枝に）。
- **`TopoNode.isNew`**: その章で構成図に加わったノードへ「追加」チップ（graph描画・ノード右上角）。「構成図が育つ」を毎章可視化する。
- **tree の幹ラベル**: `edgeLabels` に幹ペアの項目があればそのラベルを優先（`''` で非表示）。無ければ従来どおり「トランク」。
- **構成図の向き規約**: 境界が登場する第8章以降は **上=外部（インターネット）・下=内部** に統一（第8章全体図で確立）。
- **`check` ブロック／`TextbookChapter.checks`**: 本文中の「設問例」（午後節に1問）と章末の「理解の確認」（2〜3問・タップで答え表示）。全章に置く。
- **領域フォーカス（§3.9）は当面不使用**: 第7章レビューで「中途半端な省略」と差し戻し。複雑構成図は「全ノード1枚（stack N段）」を第一候補とする。
