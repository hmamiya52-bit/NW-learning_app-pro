# 教科書モード 基本設計書（v3）

最終更新: 2026-07-05（第1〜8章の実装・多角レビューの反映を統合）

本書は `docs/textbook-mode-requirements.md`（要件）に基づく基本設計であり、**現行実装（第1〜8章公開済み）の姿と、第9章以降を作るときの規範**を1冊にまとめる。図の確定仕様は `textbook-figure-spec.md`、全体マップは `textbook-curriculum-design.md`、章別の節設計は `textbook-chapter-designs.md` が正。

## 1. プロダクトの形

- **読み物＋1画面図解ユニット＋確認問題**。章は上から読む読み物で、図解は「図＋説明＋操作＝1画面」の単位。読むだけで終わらせず、章末に想起の機会（理解の確認）を置く。
- **1つのネットワーク構成図が章を追うごとに育つ**。第1章の最小構成（PC—L2SW—ルータ—Web）が、第20章で午後相当の全体図になる。学習者は同じ図の成長を見続けることで午後の構成図に見慣れる。
- **差別化は「動き」**。紙の教科書にできない、ステップ式の有限アニメーション（パケットが構成図を旅する・表が埋まる・経路が切り替わる）を主役にする。

### 章ページの構成（実装済みの現行形）

1. ヘッダ（章番号・タイトル・summary・所要時間・既読ボタン）＋ intro
2. **この章の内容**（節ミニ目次。折り畳み・節アンカーへリンク）
3. 節 ×4〜7（短い本文 ↔ 図 ↔ callout。午後節には**設問例**を1問）
4. **理解の確認**（章末チェック2〜3問。タップで答えを開く）
5. この章で持ち帰る考え方（takeaways）
6. 既読ボタン・前後章ナビ・一覧へ戻る

## 2. 設計原則（不変・全図共通）

1. **図データは座標を持たない論理データ**。トポロジ（ノード・リンク・ゾーン）と見せる順（ステップ）だけを持ち、レイアウトはコンポーネントが画面幅から算出する。固定px幅・`min-width` による横スクロール前提は禁止（初代実装の失敗の根本原因）。
2. **スマホ最優先**。375/360px で横スクロール0。図＋説明＋操作が概ね1画面。本文は14〜15px、図中ラベルも10px以上を保ち、縮小で文字を潰さない。
3. **アニメは有限**。無限ループ（SMIL `indefinite` 等）は禁止。ループやストームも2〜4ステップの有限表現にする（スクリーンショット・DOM検証を可能に保つ意味もある）。
4. **操作ボタンは動かない**。説明文・ヘッダ表・sideTable・図ステージを固定高にし、ステップを送ってもボタン位置が変わらない。sideTable は**行数をステップ間で固定**し、未記録は「—」で埋める。
5. **図は1図1理解単位**。詰め込むくらいなら分割かステップ化。逆にスカスカで意味の薄い図は載せない。
6. **各図で最も分かりやすい形を優先**する。コンポーネント共通化のために、一度合意した見せ方を崩さない（ループは曲線、OSPFは三角形、経路表は全ノード縦積み、など）。
7. `prefers-reduced-motion` では移動を抑えハイライトのみ。

## 3. データモデル（`src/data/textbook/types.ts` が実体）

```ts
interface TextbookChapter {
  id / order / title / summary / status('published'|'draft') / estimatedMinutes
  intro: Block[]
  sections: { heading: string; blocks: Block[] }[]
  takeaways: string[]
  checks?: CheckItem[]          // 章末「理解の確認」（2〜3問）
}

type Block =
  | { kind: 'text'; text: string }
  | { kind: 'callout'; tone: 'info'|'warn'|'tip'; title; body }
  | { kind: 'figure'; figure: Figure }
  | { kind: 'check'; label?: string; items: CheckItem[] }   // 本文中の問い（label例:'設問例'）

interface CheckItem { question; answer; note? }              // 答えはタップで開く

type Figure = PacketFlowFigure | OsiStackFigure | EncapFigure | AddressTableFigure
  | SequenceFigure | TimelineFigure | RecordTableFigure | SubnetCalcFigure | SegmentMapFigure
```

- 図の詳細スキーマ（`Topology` の `layout`/`stack`/`leafIds`/`edgeLabels`、`PacketStep` の `focus`/`headers`/`status`/`blockedLink`/`tableRows` 等）は `textbook-figure-spec.md` §3 を正とする。
- **リッチテキスト装飾 `[[blue/green/amber/red/slate/violet/rose/emerald:語]]`** が有効な場所: 本文 text・callout body・図の caption / takeaway・章 takeaways・check の question / answer / note。**無効な場所**（素のテキスト）: 図 title・ステップ explanation / note・record-table セル・timeline ラベル・topology のノード / ゾーンラベル。
- 新しい図種が要るとき: `types.ts` の `Figure` union に kind 追加 → `FigureRenderer.tsx` に分岐 → `figures/` にコンポーネント。新ロールは `NodeRole` ＋ `figureTokens.ts`（ROLE_ICON / ROLE_TONE）。**2章以上で使う見込みのものだけ共通 kind 化**する。

## 4. 図解コンポーネント（概要）

| 部品 | 役割 |
|---|---|
| `FigureFrame` | 図の共通枠（title / caption / takeaway） |
| `StepperControls` + `useStepper` | 戻る／次へ＋ドット。**ドットはタップでそのステップへジャンプ可**。説明文は `aria-live="polite"` |
| `PacketFlowFigure` | 動く図の親。chain（`TopologyView`）と graph（`GraphTopology`）を `topology.layout` で切替。ヘッダ表（L2/L3/L4＋変わる/そのまま/外すチップ）と sideTable（ステップ連動で埋まる表）を持てる |
| `TopologyView`（chain） | 座標レスの一直線。PC=横並び（ゾーン箱は伸長率=ノード数比で回線に幅を配分）／スマホ=縦リフロー。パケットは**封筒マークが線上を移動**し、ラベルは**回線中央に固定した吹き出し**（「送信元/宛先＋値」は2行組） |
| `GraphTopology`（graph・SVG 320幅） | 実トポロジ描画。tree／loop／triangle／**stack N段**の4モード＋`leafIds`。動きは線・矢印の強調で表し、チップは出さない |
| 静的・段階図 | OsiStack / Encap（任意段数の入れ子） / AddressTable（対比カード） / Sequence（ラダー） / Timeline / RecordTable（スマホ=1行1カード） / SubnetCalc / SegmentMap |
| `CheckList` | 設問例・理解の確認のカード（答えの開閉） |

使い分け・確定仕様・Do/Don't は `textbook-figure-spec.md` を参照。

## 5. 章テンプレート（現行形）

標準構成と分量の目安:

1. **intro** 2〜3ブロック — 前章からの接続（何が埋まっていないか）＋この章のねらい。第1章の伏線に触れる章はここで予告。
2. **構成図の差分の節**（topology ramp で図が育つ章のみ）— 「何が加わったか」を全体図のツアー（2〜4ステップ）で見せる。差分は**ステップの説明文**で言い切る（図中の「追加」バッジ類は不採用）。
3. **概念の節 ×2〜4** — 短い本文 → 図 → callout のリズム。1節1テーマ（2テーマ同居は節を分ける）。
4. **午後の節** — 問われ方の型を短く＋callout＋**設問例1問**（check ブロック、label '設問例'。設問文は「〜か。」の疑問形で試験の型に寄せ、命令形「答えよ」は使わない）。
5. **checks（章末・理解の確認）2〜3問** — 章の核を想起させる問い。答えは1〜2文＋補足。
6. **takeaways 4〜6点**。

- 図は1章に3〜6個、callout は3〜5個。所要時間は15〜20分に収める。
- `estimatedMinutes` の目安式: **本文文字数÷400 ＋ 動く図の総ステップ数×0.4 ＋ 静的図×1 ＋ 確認問題2、5分単位に丸め**。
- 章構成テンプレの「型崩し」（障害シナリオ起点等）は**保留中**（2026-07-04 ユーザー判断。全体バランス調整が必要なため、全20章が見えた段階で再検討）。

## 6. 文章・用語・表記（レビューで確定した全ルール）

### 6.1 トーン

- 人が隣で教えるような自然体。語りかけは過剰にしない。
- **敬体（ですます調）＋言い切れるところは体言止め。常体（〜する。／〜だ。／〜である。／〜ない。／〜い。終止）は使わない。** 対象は本文だけでなく、図のキャプション・takeaway・explanation・表のセル・タイムライン項目・カード見出し・確認問題まで**全フィールド**。
- 避ける: 「〜することができます」「〜になります」の連発、無意味な前置き、箇条書きだけで終える節。
- 1節の本文は短く。理解の主役は図と動き。同じ値を caption・本文・callout で繰り返す「循環」は差し戻し対象。

### 6.2 用語

- **ネスペ午後・応用情報に出ない用語は使わない**（例: 「5タプル」不可→「通信を見分ける5つの情報」）。迷ったら過去問の語彙。
- **造語禁止**（「出口ルータ」等）。標準語（デフォルトゲートウェイ／次ホップ／ブロードキャストドメイン…）を使い、言い換えは説明的な普通名詞に留める。
- 専門用語は初出で平易な言い換えを添える。深入りしない用語は「名前だけ押さえれば十分」と明示し、暗記の負担を与えない。
- 前方参照は「詳しくは第N章」と一言。既習参照は内容を1行再掲してから使う（「第1章で見たとおり」だけにしない）。

### 6.3 表記規約

| 項目 | 規約 |
|---|---|
| ダッシュ | **原則使わない**。とくに見出し・図タイトル・calloutタイトルの「A——B（主題——副題）」型は禁止（全章で同型が反復し強いテンプレ臭＝AI臭。2026-07-13にユーザー指摘で全章から撤去）。見出しは「BするA」「AでBする」等の自然な一句に。本文の言い換え・同格は読点／句点／「つまり」「いわば」「という」で表す（句点で切るときは直前が常体にならないよう敬体か体言止めに）。やむを得ず使う場合のみ全角ダーシ2つ「——」前後スペースなし |
| 宛先／あて先 | 図中の値・ラベル（ヘッダ表・表セル・パケット吹き出し・列名）=「宛先」／本文・説明文・caption・takeaway の地の文=「あて先」 |
| MACアドレス例示 | ハイフン区切り（00-1A-2B-00-00-A1） |
| ハンドシェイク | 「3ウェイハンドシェイク」（3wayとは書かない）。応答は「SYN/ACK」 |
| NAPT | 初出で「（IPマスカレードとも呼びます）」を併記 |
| IP:ポート | コロン表記（192.168.10.10:51000）は初出の章で一言説明 |
| 言い回し | 「押さえれば十分」等の定型句を同一章内で繰り返さない |

### 6.4 執筆後の点検（コミット前に必ず）

```
grep 常体終止: (る|だ|ない|しい|多い|速い…)。 → 全フィールド対象・コードコメントは除外
grep 非ネスペ用語・造語・定型句の章内重複・表記ゆれ（あて先 in 図中 等）
grep ダーシ「——」: 原則0。とくに見出し・タイトルの主題——副題型は禁止
※ ——を句点に置換したら直前が常体（動詞終止＋。）になりやすい→敬体か体言止め・連体（〜という/〜こと）に必ず直す
```

## 7. 品質バー（差し戻し実績と再発防止）

実際に差し戻された判断を規範化する。迷ったらここへ戻る。

| 事例 | 規範 |
|---|---|
| 領域フォーカス（俯瞰＋現在ゾーンのみ詳細）が「中途半端な省略で分かりづらい」と差し戻し（第7章） | **複雑な構成図は「全ノード1枚」（stack N段）を第一候補**。省略を使うなら本当に分かりやすいかを厳しく問い、モックで合意 |
| VLAN図の一直線が「実トポロジと違う」と差し戻し（第5章） | **構成図は実トポロジで描く**。星形・ループ・三角形は graph レイアウトで |
| 移動するラベル付きチップがノードを覆った（第8章） | **ノード・ラベルを覆う要素を作らない**。動きは線の強調＋方向矢印＋小さなマーク、情報は固定位置の吹き出しへ |
| 「追加」チップが「余計な情報」と差し戻し（第8章） | **図にメタ情報を足さない**。差分・注記はステップ説明文で伝える |
| HTTPS入れ子のencapが余白だらけで削除（第4章） | **スカスカの図は載せない**。図を足すなら密度と意味を確認 |
| 同じ値の繰り返し・常体混在・非ネスペ用語 | §6 のルールと grep 点検で防ぐ |

**新しい見せ方が要る図は、実装と同じ座標・色で show_widget のモックを作り、ユーザーと方向合意してから実装する**（スクリーンショットが使えない環境での見た目合意の代替として実績あり）。

## 8. 検証フロー（章公開前の必須チェック・実証済み手順）

dev サーバ: `preview_start`（name "nwsp-dev", port 5173）。HMR の中間状態で古いエラーが残ることがあるため、疑わしければ再起動してから確認する。ログインゲートは localStorage の `nwsp:auth` で回避。章URLは `/textbook/<id>`。

**`preview_screenshot` はこの環境ではタイムアウトするため、一次検証は `preview_eval` の DOM 計測**:

1. 横スクロール: `scrollWidth - clientWidth === 0` かつ幅超過要素0を **375 / 360 / 1280px** で（`preset:'mobile'` は375固定。360は width 明示で resize）。
2. 動く図: 全ステップを「次へ」で送り、**操作ボタンの top がステップ間で不変**。`useStepper` は同期クリックが合体するため **1クリック=1 eval** に分ける（ドットのジャンプで任意ステップへ飛べる）。
3. graph SVG: 全 `<text>` の BBox が viewBox（0..320）内。
4. chain のパケット吹き出し: ノードのアイコンとバウンディングボックスが交差しないこと（PC幅 740/760 と 375 で確認）。
5. `preview_console_logs` で warn / error 0（サーバ再起動後に確認）。
6. `tsc --noEmit` 0。
7. §6.4 の grep 点検。

目視が必要な箇所は「`/textbook/<id>` で直接確認を」とユーザーに委ねる。

## 9. 制作フロー（1章を起こす手順）

1. `textbook-chapter-designs.md` の該当章を正として**詳細設計**（節構成・各図の見せ方とデータ・構成図の育て方・スマホでの収め方）。作業メモは `docs/textbook-chNN-design.md` に残してよい（正ではない）。
2. 新しい見せ方が要る図だけ show_widget でモック→合意（§7）。
3. `chapters/chNN-xxx.ts` を作成（直近章を雛形に）。`index.ts` の `DRAFTS` から外して import・配列に追加（published 化）。**設問例1問＋章末 checks 2〜3問を必ず含める**。
4. §6.4 点検 → §8 検証 → 細かい単位でコミット＆プッシュ（メッセージは日本語）→ ユーザーへ要点報告（実装内容・検証結果・確認URL）。
5. 指摘が来たら反映。大きな方針変更はユーザー確認。

## 10. 非機能・影響範囲

- 教科書モード外へ影響を出さない（型・データ・コンポーネントは textbook 配下に閉じる）。
- 既読は専用 localStorage キー。同期・XP・活動履歴に載せない。
- 画像ファイルを追加しない（アイコンは `lucide-react`）。
- 章データは静的 import（肥大したら遅延読み込みを検討）。
