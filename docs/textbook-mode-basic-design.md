# 教科書モード 基本設計書

## 1. 目的

教科書モードは、ネットワークスペシャリスト試験の学習を始めたばかりの人が、最初のインプットとして読むための教材モードである。

既存のノートモードは、ある程度学習した人が知識事項を整理・暗記するためのモードであり、初回インプットには情報密度が高い。教科書モードでは、初学者が「何を」「なぜ」「どの順番で」理解すればよいかを、読み物と図解で丁寧に説明する。

対象読者は、応用情報技術者試験に合格済みで、インフラエンジニア経験が未経験から1年目程度の学習者とする。

## 1.1 章別詳細設計

章本文を実装する前に、章別詳細設計を作成する。

第2章以降の章順・学習ストーリー・図解方針は、次の文書を前提に見直す。

- `docs/textbook-afternoon-exam-research.md`
- `docs/textbook-curriculum-architecture.md`
- `docs/textbook-mode-research.md`

第1章の本文、図解、動く図解の実装基準は、`docs/textbook-chapter-detailed-design.md` に従う。第1章の既存実装と章別詳細設計が矛盾する場合は、章別詳細設計を優先する。

第2章の詳細設計は `docs/textbook-chapter-detailed-design.md` の第9章に従う。第3章から第20章までの詳細設計は、`docs/textbook-chapters-03-20-detailed-design.md` に従う。

各節の図解を実装するときは、`docs/textbook-diagram-detail-matrix.md` で、図の目的、図の型、構成要素、動き、午後問題での読解行動を確認してから作る。

## 2. 設計方針

教科書モードは、既存機能から独立した読み物コンテンツとして実装する。

- 読み物中心にする
- 基礎的な内容を分かりやすく説明する
- 原則として全ての節に図解を入れ、図の意味を本文で丁寧に説明する
- 静的な図だけでなく、パケットの流れを見せるインタラクティブ図解を積極的に使う
- ネスペ試験対策と実務ネットワーク理解をバランスよく扱う
- 読んでいて面白く、ためになり、続きを学びたくなる文章を目指す
- 章は順番に追加していく
- 画像ファイルは使わず、HTML/CSS/軽量なReactコンポーネントで図解する

## 3. 既存機能との分離方針

教科書モードは、他ページの学習データや機能と連携しない。

分離するもの:

- ノートモードのデータ
- ノート理解度
- クイズの正答記録
- 午後問題演習記録
- XP
- 勲章
- 活動履歴
- PC・スマホ同期
- 問題検索
- プロトコル一覧

例外として、入口導線のみ既存画面へ追加する。

- ホーム画面に教科書モードへのカードを追加する
- サイドバーに教科書モードへのメニュー項目を追加する

それ以外の既存ページは、教科書モードの状態を参照しない。教科書モード側も、既存ページの学習状態を参照しない。

## 5. 追加するモジュール

```text
src/data/textbookChapters.ts
src/lib/textbookReadState.ts
src/pages/Textbook.tsx
src/pages/TextbookChapter.tsx
src/components/textbook/TextbookDiagram.tsx
src/components/textbook/PacketFlowVisualizer.tsx
src/components/textbook/TextbookCallout.tsx
```

役割:

- `textbookChapters.ts`: 教科書モード専用の章データ
- `textbookReadState.ts`: 教科書モード専用の既読管理
- `Textbook.tsx`: 章一覧ページ
- `TextbookChapter.tsx`: 章詳細ページ
- `TextbookDiagram.tsx`: 図解表示用の共通コンポーネント
- `PacketFlowVisualizer.tsx`: パケットの流れを段階的に見せるインタラクティブ図解
- `TextbookCallout.tsx`: 補足、実務ポイント、試験ポイントなどの表示

初期実装では、コンポーネントを増やしすぎない。図解や補足表示が複数章で再利用できる段階になったら、共通コンポーネントへ切り出す。

## 6. データ設計

### 6.1 章データ

```ts
export type TextbookChapterStatus = 'published' | 'draft'

export interface TextbookChapter {
  id: string
  order: number
  title: string
  description: string
  status: TextbookChapterStatus
  estimatedMinutes: number
  intro: string[]
  sections: TextbookSection[]
  examFocus: string[]
  practicalFocus: string[]
  pitfalls: string[]
  summary: string[]
}

export interface TextbookSection {
  heading: string
  body: string[]
  diagrams?: TextbookDiagram[]
  callouts?: TextbookCallout[]
}
```

### 6.2 図解データ

図解は画像ファイルではなく、Reactで描画する軽量な構造化データとして持つ。

```ts
export type TextbookDiagramType =
  | 'layer-stack'
  | 'network-flow'
  | 'device-layout'
  | 'header-structure'
  | 'sequence'
  | 'comparison'
  | 'interactive-flow'

export interface TextbookDiagram {
  type: TextbookDiagramType
  title: string
  description: string
  points: string[]
  data: unknown
}
```

`data` は図の種類ごとに必要な構造を持つ。初期実装では、型を厳密に分けすぎず、第1章で必要な図から実装する。複数章で図の種類が固まった時点で、`LayerStackDiagramData` などへ分割する。

### 6.3 インタラクティブ図解データ

パケットの流れを疑似的に見せる図解は、教科書モードの重要な差別化要素として扱う。紙の教科書では表現しづらい「通信が順番に進む感じ」「機器ごとに見ている情報が違う感じ」を、軽いアニメーションとステップ表示で理解できるようにする。

```ts
export interface PacketFlowScenario {
  id: string
  title: string
  description: string
  nodes: PacketFlowNode[]
  steps: PacketFlowStep[]
}

export interface PacketFlowNode {
  id: string
  label: string
  role: 'pc' | 'switch' | 'router' | 'server' | 'dns' | 'firewall' | 'internet'
  hint?: string
}

export interface PacketFlowStep {
  id: string
  title: string
  from: string
  to: string
  packetLabel: string
  explanation: string
  headerFocus?: {
    sourceMac?: string
    destinationMac?: string
    sourceIp?: string
    destinationIp?: string
    protocol?: string
    port?: string
  }
}
```

初期実装では、厳密なネットワークシミュレータではなく、教材理解のための「動く通信図解」とする。`PacketFlowStep` は内部実装名であり、画面上の教材名としては使わない。

必要な操作:

- 自動再生
- 一時停止
- 次へ
- 前へ
- 最初から
- 現在ステップの説明表示

表示する観点:

- どの機器からどの機器へ進んでいるか
- 今流れているものがARPなのか、IPパケットなのか、TCPなのか、HTTPなのか
- そのステップで注目すべきヘッダ情報
- その機器が何を判断して次へ送っているか
- MACアドレスが区間ごとに変わること
- 宛先IPアドレスは基本的に最終宛先のまま進むこと

避けること:

- 本格的なCisco Packet Tracer相当のシミュレータにしない
- ユーザーに機器設定を入力させない
- ルーティングテーブルやARPテーブルを完全再現しようとしない
- アニメーションを派手にしすぎて本文理解を邪魔しない
- スマホで操作しづらいUIにしない

### 6.4 補足表示データ

```ts
export type TextbookCalloutType =
  | 'important'
  | 'practical'
  | 'exam'
  | 'pitfall'
  | 'analogy'

export interface TextbookCallout {
  type: TextbookCalloutType
  title: string
  body: string[]
}
```

用途:

- `important`: 重要ポイント
- `practical`: 実務ではどう見えるか
- `exam`: ネスペでどう問われるか
- `pitfall`: 初学者がつまずきやすい点
- `analogy`: たとえ話、理解補助

## 7. 既読管理設計

教科書モード専用の `localStorage` キーを使用する。

```text
nwsp:textbook:read_chapters:v1
```

保存形式:

```ts
type TextbookReadMap = Record<string, string>
```

例:

```json
{
  "layer1-3": "2026-06-13T10:30:00.000Z",
  "layer4-7": "2026-06-14T09:20:00.000Z"
}
```

操作:

- `getTextbookReadState()`: 既読マップを取得する
- `isTextbookChapterRead(chapterId)`: 指定章が既読か判定する
- `markTextbookChapterRead(chapterId)`: 指定章を既読にする
- `markTextbookChapterUnread(chapterId)`: 指定章を未読に戻す

注意点:

- 既読状態は同期対象にしない
- 活動履歴へ記録しない
- XPや勲章を付与しない
- 設定画面の全データリセットに含めるかは別途判断する

## 8. 章構成

章立ては、ノートモードのカテゴリを素材にしつつ、初学者が読む順番と午後問題で求められる読解力から逆算して再編する。

正式な学習順・章間依存は `docs/textbook-curriculum-architecture.md`、章ごとの詳細設計は `docs/textbook-chapter-detailed-design.md` と `docs/textbook-chapters-03-20-detailed-design.md` を優先する。

第21章、第22章は作らない。補足したい内容が出た場合も、既存の20章の中に統合する。

- 公開済み章は通常カードで表示し、詳細ページへ遷移できる
- 次に公開予定の1〜2章は「準備中」として薄く表示する
- それ以降の未作成章は、折りたたみ可能なロードマップとして表示する
- 未作成章へ直接アクセスされた場合は、章一覧へ戻れる準備中画面を表示する
- 公開順は必ず章番号順にする。内部実装が前後しても、途中の章を飛ばして公開しない

## 9. 各章の標準構成

各章は、次の流れを基本とする。

1. 導入: この章は何の話か
2. まずイメージで理解する
3. 図で全体像を見る
4. 仕組みを順番に読む
5. 実務ではこう見える
6. ネスペではこう問われる
7. 初学者がつまずきやすいところ
8. まとめ: この章で持ち帰る考え方

ノートモードのように分野別に整理しつつ、本文は暗記カードではなく、初学者向けの読み物として構成する。

## 10. 図解設計

画像ファイルは使わない。

図解は、HTML/CSS/Reactコンポーネントで描画する。これにより、表示が軽く、レスポンシブ対応しやすく、本文データと一緒に管理できる。

使用する図の種類:

- レイヤ構造図
- 通信フロー図
- 機器構成図
- ヘッダ構造図
- シーケンス図
- 比較表
- 動く通信図解

図解の表示ルール:

- 原則として各節に最低1つの図解を置く
- 1つの図に情報を詰め込みすぎない
- 図の直後に説明文を置く
- 「この図は何を表しているか」を明記する
- 「どこを見ればよいか」を説明する
- 「誤解しやすい点」を必要に応じて補足する
- ネスペ午後問題での読み方につながる説明を入れる

図解の詳細設計で必ず決めること:

- 何を説明するための図か
- 読者にどの誤解を解いてほしいか
- 午後問題風構成図、シーケンス図、ヘッダ構造図、比較表など、どの図の型で見せるか
- 図に置く機器、ネットワーク境界、ヘッダ、メッセージ、ログ、観測点
- 強調色を使う対象と、薄く表示する対象
- 動く図解の場合、各ステップで何が移動し、どのラベルやヘッダを強調し、説明文がどう変わるか
- 図の直後に置く本文で、どの順番で読み解かせるか

表示品質の完了条件:

- 線、矢印、ラベル、機器名が重ならない
- 重要な文字が枠やバッジに隠れない
- 320px、768px、PC幅で横にはみ出さない
- 動く図解の操作ボタン位置が説明文量で不自然に上下しない
- 黒一色の図にせず、境界、通信経路、注目ヘッダを控えめな色で区別する
- ブラウザで実際に表示し、スクリーンショットで確認してから完了扱いにする

初期実装で必要な図:

- OSI参照モデルの7階層を示すレイヤ構造図
- L4、L3、L2のヘッダが順に付くカプセル化図
- PC、L2SW、ルータ、Webサーバがどのヘッダを見るかを示す構成図
- MACアドレス、IPアドレス、ポート番号の役割比較表
- ARPの流れを示すシーケンス図
- VLANでブロードキャストドメインが分かれる図
- PCからWebサーバへ通信するまでの動く通信図解

## 11. インタラクティブ教材設計

教科書モードでは、静的な文章と図だけではなく、パケットの流れを動きで見せる教材を積極的に使う。

目的:

- 初学者が通信の順番を直感的に理解できる
- 機器ごとの役割の違いを目で追える
- ARP、L2転送、L3ルーティング、TCP接続、HTTP通信などを段階的に説明できる
- 紙の教科書では伝えにくい「パケットが進む感じ」を表現できる
- 読者が「もっと見てみたい」と感じる学習体験にする

初期実装の方針:

- 第1章に1つ、軽量な動く通信図解を入れる
- 最初の題材は「PCがWebサーバへアクセスするまで」とする
- 自動再生と手動ステップ送りの両方を用意する
- アニメーションはCSS transition中心で実装する
- 画像ファイルは使わない
- 章本文の直後に、動きの意味を文章で説明する

第1章で扱うステップ例:

1. アプリケーションデータにL4ヘッダが付く
2. L3ヘッダが付いてIPパケットになる
3. デフォルトゲートウェイのMACアドレスを知るためにARP要求を送る
4. PCがWebサーバ宛てIPパケットを、ルータ宛てEthernetフレームに入れて送る
5. L2SWが宛先MACアドレスを見てルータ側へ転送する
6. ルータが受信フレームをデカプセル化し、宛先IPアドレスを見て次ホップを決める
7. ルータが次のリンク用のL2ヘッダを付け直して再カプセル化する
8. Webサーバ側でL2、L3、L4の順に処理され、アプリケーションデータが渡される

この教材で特に伝えること:

- 各リンク上ではEthernetフレームとして運ばれ、その中にIPパケットやTCPセグメントが入る
- 「IPアドレスで最終目的地を示す」と「MACアドレスで同じリンク上の次の相手へ届ける」は別の話である
- L2SWは主にMACアドレスを見る
- ルータは受信時のL2を外し、IPヘッダを見て、次のリンク用のL2を付け直す
- 別ネットワークへ行くとき、最初の送り先はWebサーバではなくデフォルトゲートウェイである
- パケットの中身を全部覚えるより、どの層の情報で次の動きが決まるかを見る

## 12. 画面設計

### 12.1 章一覧ページ

レイアウト:

- 最大幅はノートモードと同程度にする
- 上部に教科書モードの説明を置く
- 章カードを縦またはグリッドで表示する
- 公開済み章はクリック可能
- 次に公開予定の準備中章だけ薄いカードで表示する
- それ以外の未作成章はロードマップとして折りたたむ
- 既読章には「既読」バッジを表示する

章カードに表示するもの:

- 章番号
- 章タイトル
- 説明
- 読了目安
- 公開状態
- 既読状態

ロードマップに表示するもの:

- 章番号
- 章タイトル
- 一言で分かる到達目標
- 公開予定ステータス

### 12.2 章詳細ページ

レイアウト:

- 本文を読みやすい幅に制限する
- 図解は本文幅いっぱいに表示する
- スマホでも横スクロールに頼りすぎない
- 長い章でも見出しで呼吸できる構成にする

主なUI:

- 章一覧へ戻る
- 前の章
- 次の章
- 既読にする
- 未読に戻す

詳細ページでは、テスト、理解度ボタン、XP表示、勲章表示は出さない。

インタラクティブ教材のUI:

- 図の上部にシナリオ名を表示する
- 中央に機器と通信経路を表示する
- 現在流れているパケットを線上で動かす
- 下部に現在ステップの説明を表示する
- 操作ボタンは「戻る」「再生/停止」「次へ」「最初から」を基本とする
- スマホでは縦方向に見やすい配置へ切り替える

## 13. 教材品質方針

教科書モードの価値は、単に知識を並べることではなく、初学者がネットワークの見方を獲得できることにある。

文章方針:

- いきなり用語定義から入らない
- 現実の通信イメージから入る
- 「なぜ必要なのか」を説明する
- 応用情報の知識をネスペ水準へ橋渡しする
- 実務で起きそうな場面を混ぜる
- ネスペ午後問題の読み方につながる視点を入れる
- 暗記よりも、構造が見えることを重視する
- 読後に「少し分かった」「次も読めそう」と感じられる締め方にする

避けること:

- 用語の羅列
- 過度に専門的な省略
- 図だけ置いて説明しないこと
- ノートモードと同じ粒度の暗記メモ化
- 試験対策だけに寄せすぎること
- 実務雑談だけに寄せすぎること

## 14. 第1章「OSI参照モデルと通信の全体像」の教材設計

### 14.1 章の狙い

第1章は、個別技術の暗記ではなく、通信を階層で見るための地図を作る章である。最初にOSI参照モデルを示し、その中でL4、L3、L2、L1の役割、カプセル化、デカプセル化、再カプセル化を理解させる。

詳細な節構成と図解仕様は `docs/textbook-chapter-detailed-design.md` を優先する。

### 14.2 想定セクション

1. OSI参照モデルは通信を見るための地図
2. データには下位層へ渡すたびにヘッダが付加される
3. クライアント、ネットワーク、サーバは同じデータを違う見方で扱う
4. MACアドレス、IPアドレス、ポート番号の役割
5. ARPはL2フレームを作るために必要になる
6. PCからWebサーバへ届くまでの動く図解
7. VLANとブロードキャストドメイン
8. ネスペ午後では構成図の境目を読む

### 14.3 使用する図

- OSI 7階層の縦積み図
- L4/L3/L2ヘッダが順に付くカプセル化図
- PC、L2SW、ルータ、Webサーバが見るヘッダの違いを示す構成図
- MACアドレス、IPアドレス、ポート番号の対応表
- 同一ネットワーク宛てと別ネットワーク宛てのARP比較図
- VLANで分かれたブロードキャストドメインの図
- PCからWebサーバへ通信するまでの動く通信図解

### 14.4 読者に持ち帰ってほしいこと

- OSI参照モデルは暗記表ではなく、通信を分解して見るための地図である
- TCPセグメント、IPパケット、Ethernetフレームは入れ子関係にある
- MACアドレスは同じリンク上の次の相手、IPアドレスは最終的な通信相手、ポート番号は端末内のサービスを示す
- ルータではL2ヘッダが付け替わるが、IPヘッダの宛先IPは基本的に維持される
- ネスペ午後の構成図では、L2境界、L3境界、ブロードキャスト範囲を読む

## 16. 実装順序

教科書モードの画面・既読管理・入口リンクの基盤を先に作る。その後は章ごとに次の順番で進める。

1. 対象章の詳細設計を確認する
2. `docs/textbook-diagram-detail-matrix.md` を参照し、各節の図解について、目的、図の型、構成要素、動き、本文での読み解き順を固定する
3. 本文を実装する
4. 静的図解または動く図解を実装する
5. PC、タブレット幅、スマホ幅で表示確認する
6. 技術的正確性、日本語、話の流れ、図解品質をセルフレビューする
7. 既存ノートモード、クイズ、同期に影響がないことを確認する
8. コミットし、pushする

第1章から第20章まで、章単位でこの工程を繰り返す。章をまたいだ共通コンポーネント化は、2章以上で同じ図解型が安定してから行う。
公開は章番号順に行う。内部実装の都合で後続章の部品を先に作った場合でも、読者へは未公開のままにする。

## 17. 確認観点

機能確認:

- `/textbook` が表示できる
- 公開済み章だけ詳細へ進める
- 準備中章とロードマップの扱いが分かりやすい
- `/textbook/layer1-3` が表示できる
- 既読にする / 未読に戻す が動作する
- リロード後も既読状態が残る
- 前後章ナビゲーションが動作する
- 動く通信図解の再生/停止/前へ/次へが動作する
- 動く通信図解のステップ説明が現在位置と一致する

分離確認:

- ノート理解度が変化しない
- XPが増えない
- 勲章が増えない
- 活動履歴に記録されない
- 同期データに含まれない
- 問題検索結果に教科書本文が混ざらない

表示確認:

- PCで本文が読みやすい
- スマホで文字と図がはみ出さない
- 図解が軽く表示される
- 動く通信図解がスマホでも操作しやすい
- アニメーションが本文の読みやすさを邪魔しない
- 長い文章でも見出しで読み進めやすい
- 章一覧で既読・未読・準備中・ロードマップが判別できる
- 図解の線、矢印、ラベル、機器名が重ならない
- 重要な文字が枠、バッジ、吹き出しで隠れていない
- 動く図解のボタン位置がステップ説明の長さで不自然にずれない

教材確認:

- 初学者向けの説明になっている
- 用語定義だけで終わっていない
- 図の意味が本文で説明されている
- 動く教材の意味が本文で説明されている
- 試験対策と実務理解の両方に触れている
- ノートモードと役割が重複しすぎていない
- 図解ごとに「何を説明する図か」「どこを見る図か」「何を誤解しないための図か」が本文で分かる

## 19. 引き継ぎ時点の状態と裁量

2026-06-21時点で、教科書モードは第1章から第20章までの「図解先行」状態で実装されている。各章ページ、工事中表示、既読管理、章一覧、基本図解、動く図解、読解の足場図は存在する。ただし、これは完成版教材ではない。本文説明文、章固有の専門図、章ごとの精密レビュー、読者に読ませる文章としての完成度は後続工程で作り込む。

現時点の実装は、次の性格を持つ。

- 第1章から第20章まで、`status: 'diagram'` の章として表示できる
- 各章に、構成図、見る観点を分ける図、動く手順図、読解の足場図がある
- 図の技術正確性について、DNS/DHCP/ARP、SSO、メール、VoIP、IPv6、Web、運用監視の代表的な誤りは修正済み
- 390px、768px、1280px幅で、全章の主要図解が横スクロールなしで表示できることを確認済み
- 本文は未完成であり、画面上の短い説明や図内ラベルは教材本文の代替ではない

引き継ぎ後の担当者には、次の裁量を認める。

- 教科書モード内に閉じる範囲で、本文、図解データ、図解コンポーネント、章内セクション構成を修正してよい
- 技術的に不正確、読みにくい、初学者向けでない図解は、既存実装に遠慮せず作り直してよい
- 同じ図に詰め込むと読みにくい場合は、基本図解と増補図解に分割してよい
- 各章の本文実装時に、章固有の専門図を追加してよい
- 図解コンポーネントは、2章以上で同じ型が安定してから共通化してよい
- 技術判断に不安がある場合は、IPA公式の過去問題・講評、RFC、主要ベンダドキュメントなど一次情報を確認してよい

ただし、次の制約は維持する。

- 教科書モードは、ノートモード、クイズ、午後問題演習、同期、XP、勲章、活動履歴と連携しない
- 既存ページへの変更は、ホームとサイドバーの入口導線を除いて行わない
- 画像ファイルを追加せず、SVG、HTML、CSS、Reactで図解する
- 章順は、`docs/textbook-curriculum-architecture.md` の全20章体系を基準にする
- 章を公開扱いにする前に、技術正確性、日本語、話の流れ、表示品質をセルフレビューする
- 作業単位ごとに、実装計画書へメモを残し、commit / push まで行う

Claude Codeへ引き継ぐ場合は、`docs/textbook-claude-code-handoff.md` の引き継ぎプロンプトを開始プロンプトとして使う。
