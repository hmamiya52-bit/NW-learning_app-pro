# 教科書モード 要件定義書（再定義版 v1）

作成日: 2026-06-21

## 0. この文書の位置づけ

教科書モードを「要件定義からやり直す」ための文書である。

- 先行実装（codex 版）は、全章を機械生成した「本文ゼロの図解ギャラリー」であり、構成図・動くパケット図が **スマホで横スクロール必須** で、かつ **本文の日本語も不自然** だったため破棄する。
- codex 版は **本文・図解・設計書とも基本的に破棄** する（部分的な本文の流用もしない）。先行の20章カリキュラムの題材・順序は **参考程度** に留め、章構成は段階的複雑化（§5.4）を軸に基本設計で組み直す。
- 本書が固まったら基本設計を作成する（進め方は §9）。

## 1. 目的と対象読者

### 1.1 目的

ネスペ学習を始めたばかりの人が、**最初のインプット**として読む教材を提供する。既存ノートモードは整理・暗記用で初回インプットには密度が高い。教科書モードは「何を・なぜ・どの順で」理解するかを、読み物と図解で丁寧に作る。

### 1.2 対象読者

- 応用情報技術者試験に合格済み
- インフラエンジニア未経験〜1年目程度
- OSI 7階層の名前は見たことがある程度。ヘッダの付く順、MAC/IP/ポートの使い分け、各機器が何を見て動くかは曖昧

## 2. スコープ

### 2.1 やること

- 読み物中心の章コンテンツ（基礎を分かりやすく、面白く、続きを読みたくなる）
- 図解を多用し、各図が何を意味するかを本文で丁寧に説明する
- 疑似パケットトレーサー的な「動くパケット図」を入れる
- 既読管理（既読／未読）
- 章ごとにページ分割し、章は順番に追加する
- 試験対策（ネスペ午後の読解）と実務理解をバランスよく扱う

### 2.2 やらないこと

- 学習進捗・テスト・採点・スコア
- ノート／クイズ／午後演習／XP／勲章／活動履歴／同期との連携
- 画像ファイルの使用（図解は HTML/CSS/SVG/React で描画）

## 3. 他機能からの分離

- 教科書モードは他ページの学習データ・機能と **一切連携しない**（与えalso受けない）。
- 例外は入口導線のみ：ホーム画面のカードとサイドバーのメニュー項目（既存どおり）。
- 既読状態は教科書モード専用の `localStorage` キーで持ち、同期・活動履歴・XP に含めない。

## 4. コンテンツ方針（文章）

- いきなり用語定義から入らず、現実の通信イメージ → なぜ必要か → 仕組み、の順で書く。
- 応用情報の知識をネスペ水準へ橋渡しする。暗記より「構造が見える」ことを重視。
- 実務で起きる場面（障害切り分けなど）と、ネスペ午後の読み方を織り込む。
- 読後に「少し分かった」「次も読めそう」と感じる締め方にする。
- 図だけ置いて説明しない、を禁止する。図の直後に「何を表す図か／どこを見るか／誤解しない点」を本文で示す。
- **「AI が書いた感じ」を極力なくす。** テンプレ的・冗長・機械的な言い回し（「〜することができます」「〜になります」の多用、箇条書きの羅列だけ等）を避け、人が隣で教えてくれるような自然な日本語にする。
- **淡々とした説明にしない（退屈の回避）。** 文章は要点を短く言い切り、細部は図解で見せる。説明を文章で盛るより、図解の数と動きで理解させる。

## 5. 図解方針（本プロジェクトの最重要要件）

### 5.0 図解の基本姿勢（動き優先・図多め）

- **紙の教科書では出せない「動き」に注力する。** 静的な図の羅列ではなく、動的・インタラクティブな図解（動くパケット図、段階表示、ハイライト遷移、前後比較）を主役にする。これが教科書モードの差別化要素。
- **図解は多めに置く。** 長い説明文に頼らず、図の数と動きで理解させる（淡々とした文章の回避）。
- 1つの図に詰め込みすぎない。1図＝1理解単位に分割し、動きで順を追わせる。

### 5.1 共通原則（スマホ最優先）

- **スマホ（〜375px 想定）で横スクロール禁止。** 構成図・動くパケット図・表を含め、横スクロールが必要な図は不可とする。
- **図解＋その説明は、できる限り1画面に収める。縦スクロールも極力減らす。**（1図＝1理解単位）
- 文字が潰れない。線・ラベル・バッジ・機器名が重ならない。
- PC では横方向に余裕を持って表示し、スマホでは縦リフロー／ホップ単位フォーカスで成立させる。
- 黒一色にせず、境界・経路・注目ヘッダを控えめな色で区別する。

### 5.2 動くパケット図（疑似パケットトレーサー）の要件

第1章で確認したモックアップの方向を採用する。

- **全体のネットワーク構成図を常に表示する**（縮小して全機器が見える状態）。構成図に見慣れさせることが目的。
- **いま説明している区間（ホップ）を構成図上で光らせる。** パケットが実際の経路上を流れる「パケットトレーサー感」を出す。
- 区間ごとに、L2（MAC）／L3（IP）／L4（ポート）の現在値と、その意味を併せて示す。特に「**宛先 MAC は区間ごとに変わる／宛先 IP は最終宛先のまま**」が一目で分かること。
- ルータ／L3SW での付け替え（デカプセル化→経路判断→再カプセル化）、受信側のデカプセル化をステップで見せる。
- 操作：戻る／再生（自動巡回）・停止／次へ。現在ステップの説明と構成図のハイライトが一致する。
- 上記すべてを **スマホ1画面・横スクロールなし** で成立させる。

### 5.3 静的図の要件

- OSI 7層、カプセル化（ヘッダが順に付く）、フレーム/パケット/セグメントの入れ子、MAC/IP/ポート対応表、ARP（同一NW宛て/別NW宛て）、VLAN とブロードキャストドメイン等。
- いずれも §5.1 の共通原則（スマホ1画面・横スクロールなし・可読）に従う。スマホでは縦方向に積み替えるなどしてレイアウトする。

### 5.4 ネットワーク構成図の段階的複雑化（章を貫く設計方針）

- 第1章は **最小構成**（例：PC—L2SW—L3SW—Webサーバ）でよい。
- 章が進むにつれて構成図を **段階的に複雑化** し、ゾーン（境界）・冗長化・複数経路・複数セグメントなどを少しずつ増やし、最終的にはネスペ午後問題相当の構成図を読めるところまで **見慣れさせる**。
- 複雑化しても §5.1 を崩さない。複雑な構成図は「縮小した全体図（俯瞰）＋注目領域/ホップのフォーカス」「段階的開示」で、スマホでも横スクロールなしに読めるようにする。
- したがって図解コンポーネントは、**単純〜複雑な構成図までスケールしてもスマホで成立する**ことを設計要件とする。

## 6. 章構成・情報設計

- 情報設計は「**読み物＋1画面図解ユニット**」を採用する（ユーザー確定）。
  - 章は上から読む読み物。各図解はモックのように「図＋説明＝1画面」の単位。動くパケット図は完全ステップ式。節間は多少の縦スクロールを許容する。
- 章ごとにページを分け、公開は章番号順。未完成章は「準備中／工事中」表示。
- 各章の標準構成（目安）：導入 → イメージ → 全体図 → 仕組みを順に → 実務での見え方 → ネスペでの問われ方 → つまずき → まとめ。
- 章の並び・題材は先行20章体系を **参考程度** にしつつ、§5.4 の段階的複雑化を軸に基本設計で組み直す。本文・図解はすべて新規に作る。

## 7. 表示・品質要件（提出前チェック）

- スマホ375px：横スクロールなし、図＋説明が極力1画面、文字可読、重なりなし。
- タブレット768px・PC1280pxでも崩れない。
- 動く図の操作ボタン位置が説明文量で不自然に上下しない。
- **検証手段を先に用意する。** 前回は dev/preview サーバで `preview_screenshot` が HMR/PWA により常時タイムアウトし、スマホ表示を目視確認できないまま進めた。実装着手前に、スマホ幅の描画を確実に確認できる手段（PWA/HMR を切った確認用ビルド、または別の検証フロー）を整える。

## 8. 技術正確性基準（図・本文共通）

- L2/L3/L4/L7 の役割を混同しない。
- 各リンク上では L2 フレームとして運ばれ、その中に IP パケット・TCP/UDP セグメントが入る前提を崩さない。
- ルータ／L3SW では L2 を外し、IP を見て、次リンク用に L2 を付け替える。宛先 IP は基本的に維持（NAT 等の例外は該当章で扱う）。
- L2SW は MAC、ルータ/L3SW は IP、FW/LB は条件（ポート等）— 各機器が見る情報を混同しない。
- ARP は「L2 フレームを作るために次ホップの MAC を解決する」文脈で出す。IP パケットではなく ARP として運ぶ。
- 不安がある場合は IPA 公式（過去問・講評）、RFC、主要ベンダ資料など一次情報を確認する。

## 9. 進め方

1. 本要件定義書を確定する（←今ここ）
2. 基本設計を作成する（図解コンポーネントのデータ構造、スマホ/PC のレイアウト規則、動くパケット図の仕様、章テンプレート、検証フロー）
3. スマホ描画の検証手段を整える
4. **第1章だけ**を実装して確認 → レビュー
5. 第1章で本文トーン・図解・スマホ表示の型を固め、§5.4 に沿って順次後続章へ展開
